AISEP / 定位02 / 17

为什么需要

AI 写代码已经很快，但企业级数字化交付还缺四件事。

!会话无法沉淀

新项目从零开始，前一个项目踩过的坑不会自然进入下一个项目。

G缺少阶段闸门

一次性产出太多，reviewer 容易只看结论，风险被推迟到上线前。

D真实工程是 DAG

企业系统常有并行实现、合并、返工和复验，不是线性对话。

A架构决策太隐蔽

数据模型、协议、运行时这类高搬迁成本决策，必须在编码前被单独确认。

三大企业用场

覆盖数字化交付的 3 类典型工作：绿地、改造、逆向。

1 快速交付

绿地新需求

从 需求文档 / OKR 任务 启动，AISEP 一条龙跑 intake → integrate，产出可 PR 的 patch + 测试 + 评审 + 集成日志。

输入：seed.txt + requirements.yaml
输出：10 stage artifacts + patch.diff + report.html
实测：~17 min wall（Pilot-10b 真业务 dogfood）
2 快速重构

既有系统改造

指向 已有 codebase，AISEP 跑 architecture 阶段产出当前结构图 + risks + 重构 ADR + 渐进迁移 plan。

输入：已有项目目录 + 改造目标
输出：architecture/brief.md + ADR-NNN + risks.yaml + 切片 plan
价值：不动一行代码，先把方案锁住
3 工程化逆向

遗留代码解构

对着 无文档遗留系统，AISEP 反向产出 requirements.yaml + ADR + ZOD schema + RISK trace，把代码反推成可读的工程文档。

输入：遗留代码 + 关键路径目录
输出：REQ → ADR → ZOD → RISK 反向 trace + audit 报告
场景：合规审计 / 接管旧系统 / 离职交接（Gartner 2026 预测 40% 现代化项目用 AI 逆向，vs 2023 <10%）

企业 audit 价值

2026 SOC2 + ISO/IEC 42001 + EU AI Act 把 AI 工具 audit trail 写进硬要求。

审计要求 2026 标准要什么

  • Who/What/When/Where/Why：每个 AI 动作都要可追溯
  • immutable log + 加密完整性：日志不可篡改
  • artifact 与决策的双向 trace：需求 ↔ 实现 ↔ 测试 ↔ 风险闭环
  • Human approval gate：SOC2 CC1.4 要求关键决策有人工签字
  • Prompt-level replay：NIST AI RMF GOVERN 1.5 要求 AI 输入可重放
  • EU AI Act Article 14：高风险系统强制 audit trail（2026-08-02 deadline）

AISEP 已有 不用额外做的部分

  • 每 attempt 全字段记录：state.json 含 invocation / argv / promptHash / startedAt / endedAt
  • trace_id 链：REQ → ADR → ZOD → RISK → PATCH → verify_check
  • HITL 6 锚点：architecture / review / memory-promote / retrospect / dispatch / ask-user 全留痕
  • Forensic prompt replay：.aisep/tmp/task-<stage>-<ts>.md 渲染后 prompt 留档
  • R11 trust 边界：防 MINJA 类 prompt injection (98.2% 成功率攻击)

Fan-out

复杂功能可拆 2-4 个子任务并行实现，再统一 verify。

适合 backend / frontend / tests 这类天然能拆分的任务。
并发上限保守 4（对齐 Anthropic 后端 burst limiter，自动 retry 30s/60s/120s backoff）。
子补丁汇总成 patch_set，再统一进入 verify 阶段。
SIGTERM → 5s → SIGKILL cancel：一个 child 失败立即终止其他 siblings。
