## Findings

### B1 [BLOCKER] “零 CLI 负担”与“自动跑命令/自动 commit”冲突

**Where**: `/tmp/steward-v0-cross-prompt.md:291-297`, `427-430`, `467-486`, `488-505`

**Lens**: 1 / 5

**Issue**: G1 写的是“用户从不在终端敲命令；纯文本 prompt 粘到 Claude 窗口里”，但后续 contract 要求 Claude 自动跑 `pnpm eva:sessions`、`git worktree add`、编辑 `eva.json`、`git commit`、`pnpm eva:hook pre-remove`。

**Why this is a blocker**: 这是 Phase 0 的用户接口契约。如果不澄清，未来 session 会对“只是推荐下一步”还是“可以直接改文件并提交”产生不同解释，正好破坏 G3/G4 想解决的跨 session 一致性。

**Suggested fix**: 把 Phase 0 明确拆成两类操作：

- 纯读短语：`看 backlog 推荐下一步`、`现在哪些窗口在干啥`
- 写入短语：`开始干`、`收线`、`加待办`、`blocked`

并补一句：Phase 0 允许 Claude 代执行 repo 内命令，但用户不直接敲 CLI；所有写入和 commit 前必须展示 diff 并确认。或者反过来，把自动 commit / worktree 全部降级到 Phase 1。

### M1 [MAJOR] “每次状态转移 commit 一次”会和现有脏工作区冲突

**Where**: `/tmp/steward-v0-cross-prompt.md:430`, `485`, `504`, `507-514`

**Lens**: 1 / 4

**Issue**: contract 要求 backlog 状态转移自动 commit，但没有规定当前 repo 已有 unrelated dirty changes 时怎么办。

**Why this matters**: 当前仓库本身就可能有 `project.pbxproj` 修改、未跟踪 docs 等。自动 commit 很容易把用户其他改动一起提交，或因为 staged 状态不清导致失败。

**Suggested fix**: 加一个 invariant：

`Steward 只允许 stage/commit docs/BACKLOG.md、docs/STEWARD_PROMPTS.md、CLAUDE.md 中本次 steward 改动；若 git status 存在 unrelated staged changes，必须停止并让用户确认。`

### M2 [MAJOR] `status` 与分区存在双重 source of truth

**Where**: `/tmp/steward-v0-cross-prompt.md:324-388`, `390-406`, `507-514`

**Lens**: 1 / 2

**Issue**: 同一个任务状态同时由 section 位置（Active / Blocked / Done）和 YAML 字段 `status` 表示。

**Why this matters**: 一旦 Claude 把 `status: done` 的 item 留在 Active 段，或者把 `status: blocked` 放到 Done 段，后续 session 不知道以哪个为准。

**Suggested fix**: 明确优先级：`status` 是权威，section 只是展示；每次写回必须按 `status` 重新归类。或者移除 section 语义，只保留一个 fenced YAML 列表，再用状态排序。

### M3 [MAJOR] `assigned_session` 的身份定义不够稳定

**Where**: `/tmp/steward-v0-cross-prompt.md:401-404`, `459-464`, `475-485`

**Lens**: 2 / 3

**Issue**: `assigned_session` 可写 cwd、worktree name、`user-manual`，但算法又要用它判断活窗口和冲突。

**Why this matters**: cwd、branch、worktree name 是不同层级的身份。一个 cwd 可以换 branch，一个 worktree name 不一定等于路径，`user-manual` 又不是机器可验证 session。

**Suggested fix**: 拆字段：

- `assigned_kind`: `cursor-session | user-manual | external`
- `assigned_cwd`: 绝对路径，可选
- `assigned_branch`: 可选
- `assigned_note`: 人读说明

Phase 0 如果不想加字段，至少规定 `assigned_session` 必须是 cwd 绝对路径或 `user-manual`，不能混用 worktree name。

### m1 [MINOR] `Blocked / On Hold` 里 `status: blocked` 与 `depends_on` blocked 混用

**Where**: `/tmp/steward-v0-cross-prompt.md:356-375`, `455-458`

**Lens**: 1

**Issue**: 有些任务是显式 `status: blocked`，有些是 `planned` 但依赖未完成导致 blocked by dep。

**Suggested fix**: 在算法里区分 `blocked` 和 `blocked_by_dependency`，不要把依赖未完成的 planned 项移动到 Blocked 段。

### m2 [MINOR] `priority asc, size asc` 可能长期饿死大任务

**Where**: `/tmp/steward-v0-cross-prompt.md:454-464`

### m4 [MINOR] rollback 低估了 `BACKLOG.md` 的新增信息价值
