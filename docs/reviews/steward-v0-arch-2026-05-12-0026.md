## Findings

### F1. id 唯一性 + 跨表锚定字段缺失 → Phase 2 promote 不可逆 — [BLOCKER]
**Where**: §3.1 schema 表格 / §10 初始填充 / §5 ADR-lite "动 harness.db Issue 表 = 不动"
**Lens**: 架构可行性（L5 持久层 / 跨层数据流）
**Issue**:
- `id` 是 kebab-case 字符串，**没有命名空间前缀**。Phase 0 写下 `testflight-encryption-compliance` 这种 id 之后，Phase 2 要 promote 进 `harness.db` `issue` 表（[HARNESS_DATA_MODEL.md §1.3](../HARNESS_DATA_MODEL.md)）时，会发生两类冲撞：
  1. `issue.id` 在 schema 里是 `TEXT PRIMARY KEY`，没有约束格式，但 harness 期望 `randomUUID()` 或者 `<initiative>-<n>`。BACKLOG.md 直接写 `testflight-encryption-compliance` 没法保证 phase 2 还自由（同名 issue 已存在 / kebab-case 与系统其他 id 体系冲突）。
  2. §6 R5 提到"预留 `harness_issue_id` 字段"——这条只在 Risks 表里口头讲，**没进 §3.1 schema 表**。等 Phase 2 真要 bridge 时会发现 BACKLOG 项没字段可填，要么改 schema 要么手动维护映射表。
- I2 说"id 永久不变"，但 §3.5 / §3.6 流程都没说 id 怎么生成。Claude 加一项时若按 title 自动 slugify，两个 title 撞 slug 怎么办？
- Phase 0 schema 是 contract（用户明确强调"future sessions bound by this design"），**用 ADR-lite 锁死 id 生成规则**比"未来再加字段"成本低 10 倍。

**Why blocker**:
- Phase 2 promote 是 §2 G5 + §6 R5 显式承诺的演化路径，schema 字段不足会让"演化"实际变成"重写 + 数据迁移"。
- §5 ADR-lite 里"不动 harness.db Issue 表"被列为决策——但**没有显式声明 BACKLOG.md ↔ Issue 表的命名空间隔离规则**，等于把契约债务推到 Phase 2。
- 违反 HARNESS_ROADMAP §0 #3"M-1 奠基期产出最小完备契约"——schema 是契约的一部分。

**Suggested fix**:
- §3.1 schema 加 3 个必填/可选字段：
  - `id` 格式约束写死：`^[a-z][a-z0-9-]{2,49}$`（与 eva.json `name` 约束对齐）。
  - **新增 `id_namespace` 隐式约束**：BACKLOG id 一律加前缀 `bl-`（backlog），与 harness Issue 自己的 id 空间永不撞。Phase 2 promote 时新建 issue.id = randomUUID()，在 backlog 条目追加 `harness_issue_id: <uuid>`。
  - 显式 schema 字段：`harness_issue_id: string?` —— Phase 0 不填，但写进 schema 表，防止 Phase 1 加同名字段时无 SemVer。
- §5 ADR-lite 加一行 "BACKLOG.md id 永远不写入 harness.db issue.id；两套 id 空间独立。" 列入 invariants（I7）。

### F2. eva.json `worktrees[].name` 与 BACKLOG `id` / `assigned_session` 关系未定义 — [MAJOR]
**Where**: §3.1 `assigned_session` 字段定义 / §3.5 步骤 5 "git worktree add + eva.json edit"
**Lens**: 架构可行性（L5 状态分散 + 命名一致性）
**Issue**:
- eva.json 已经存在 `worktrees[].name`（参 [eva-config.ts:78](../packages/shared/src/eva-config.ts) `EvaWorktreeEntrySchema.name`）+ `status: active|done|released` 状态机。BACKLOG.md 的 task `id` 又是一个 kebab-case 串。`assigned_session` 写"cwd 或 worktree name 或 user-manual" — **三个值域**，谁是 source of truth？
- §3.5 步骤 5 让 Claude "git worktree add + eva.json edit" — 但 eva.json `worktrees[].name` 是否等于 BACKLOG `id`？看 §10 初始填充 `m2-voice-impl` 这种 id 是动态生成还是约定俗成？等 PR 流转完后 eva.json 状态机 `active→done`，BACKLOG 是否同步翻 `done`？谁先谁后？
- 现在 eva.json 已经有 `M1-mini2-stage-aware-prompts` 这种 name 形态（M2 风格 + 多词描述），BACKLOG 期望 `m2-voice-impl` 形态——两套命名约定。
- 现实风险：用户从主窗口运行 `开始干 m2-voice-impl`，Claude 写 BACKLOG 但 eva.json `worktrees[].name` 命名与之不同，下次 `pnpm eva:status` 与 BACKLOG 一对照像不像同一回事？

**Why it matters**:
- L5 持久层（[HARNESS_ARCHITECTURE.md §1 L5](../HARNESS_ARCHITECTURE.md)）的不变量是"每个写入有唯一 owner"。eva.json 是 worktree 注册表 + lifecycle 状态机；BACKLOG 自称"single source of truth"——但实际上是 BACKLOG 引用 eva.json 状态 + 命令 Claude 改 eva.json。
- 没说清"两个文件的 commit 关系"：是同一个 commit 还是分两 commit？后者会让"backlog 标 done 但 eva.json 还 active"或反向漂移。M1 retrospective 信号 #1 就是 markdown 表语义漂移导致的——同样的坑。

**Suggested fix**:
- §3.1 把 `assigned_session` 拆成两字段：`assigned_worktree: string?`（== eva.json `worktrees[].name`，可选）+ `assigned_window_label: "main" | "user-manual" | ...`（自由文本，标方位）。
- §3.5 / §3.6 显式声明"BACKLOG + eva.json 同 commit"（一个 commit 跨两文件），并把 commit msg 例子写进 STEWARD_PROMPTS.md。
- 加 invariant I7：BACKLOG `assigned_worktree` 必须对应 eva.json 中 status=active 的一行 name，否则视为不一致，下次 boot ritual 报警。

### F3. Invariant I3 "每次 status 转移 commit 一次" 在并行 session 场景会反咬 — [MAJOR]
**Where**: §4 I3 / §6 R1 / §5 ADR-lite "并发写 = single-session 假设"
**Lens**: 风险遗漏（多 agent 资源争抢）
**Issue**:
- I3 要求"每次 status 转移 commit 一次"。`dev` 分支保护强制走 PR（CLAUDE.md "Branch & Release"段），如果 BACKLOG 在 `dev` 上，每个 status flip 就是一个 PR；这显然不可行。
- 唯一可行解：BACKLOG 在主 worktree 的当前分支（feat/eva-...），跟随 PR 一起合 dev。但这违反了 §3.3 boot ritual"任何窗口都读 BACKLOG"——别的 worktree 的 feat 分支永远看不到主窗口的 BACKLOG 改动。
- §5 ADR-lite 显式声明 single-session 假设。但 §3.5 / §3.6 又允许"开新窗口"流程（说明设计预期了多窗口）。**契约自相矛盾**：单 session 假设 vs 多窗口工作流。
- 真实模式应该是：BACKLOG.md commit 直接落 `main`（或 `dev`），不走 PR。这跟 `main` branch protection 冲突。或者 BACKLOG 不 commit，靠 working-copy 同步——又违反 I3。

**Why it matters**:
- 这是 contract 的可执行性问题：dogfood 阶段第一周就会撞到（用户已经在多 worktree 跑 M1/M2 实验，eva.json 6 条记录里 5 条已经是 done）。
- HARNESS_ROADMAP §0 #11 "不引入新基础组件"——加 git merge driver 太重，是 Phase 1 才考虑的。Phase 0 必须有一个"够用"的协议。

**Suggested fix**:
- 把 `docs/BACKLOG.md` 改为**主分支直接 push 允许**（在 git branch protection 例外列表里加这一个文件路径，或者 pre-commit hook 放行 BACKLOG-only commit）。
- 或者更稳：BACKLOG.md 移到 `~/.vessel/backlog.yaml`（out-of-repo, per-machine, gitignored）—— §5 ADR-lite 显式拒绝了这个方案，理由是"跟设计文档同 PR review"。**这条理由在 Phase 0 是过度**：Phase 0 只有 1 个用户、1 台机器；"跨机器同步"是 Phase 2 需求。建议 Phase 0 用 out-of-repo，Phase 1+ 才进 repo。
- 如果坚持 in-repo：§5 ADR-lite + I3 改成"每次 status 转移**写一次**（push 视情况；连续多次 status 可以攒一次 push）"，并显式说"BACKLOG 走 main 直 push"，与 git branch protection 协同更新。

### F4. §3.4 "下一步推荐算法" 把 eva:sessions 接入了 boot ritual — [MAJOR]
**Where**: §3.3 步骤 3 / §3.4 步骤 5
**Lens**: 架构可行性（L7.2 可观测性接口稳定性 + Phase 0 KISS 原则）
**Issue**:
- `pnpm eva:sessions` 是 [scripts/eva-sessions.mjs](../scripts/eva-sessions.mjs)：解析 `ps -axo` + `~/.claude/projects/*` jsonl mtime。这是**纯派生视图**，没有稳定输出契约。
- Boot ritual 每次都跑它（每个 session 启动都 spawn 一次 ps + readdir + git branch — 在 ~/.claude/projects/ 大目录下可能 100+ 子目录），对 Cursor 启动延迟有影响（Claude 在 boot 里串行跑 grep + 文件 IO + git）。
- 它输出是 ANSI 表格 + 自由格式 — 没有 JSON schema、没有 `--format json` flag。Claude 解析靠 prompt 体察。如果 user 升级 macOS 后 `ps -axo` 字段变了，BACKLOG 算法直接挂。
- §3.4 步骤 5 用 eva:sessions 数据做"开新窗口/本窗口直接做" — 这是**契约依赖**，已经把 eva-sessions.mjs 的输出格式提升成了实质 API。

**Why it matters**:
- 契约审视角：[HARNESS_ARCHITECTURE.md §1 L2](../HARNESS_ARCHITECTURE.md) 规定 API 字段 additive + minClientVersion。这里实际上把一个工具脚本提升成 API 但没标 minClientVersion。
- L7 横切的 telemetry 系统已有 `~/.claude-web/telemetry.jsonl`，更适合做"活窗口"——但不是 Phase 0 的事。
- Boot ritual 越重越容易被忽略；§3.3 步骤 1-4 已经 4 步，第一周用户会因为这套流程慢而绕过。

**Suggested fix**:
- §3.3 boot ritual 改 lazy：步骤 1+2 保留（读 BACKLOG + stale 检查），**步骤 3 移到"用户问下一步时再跑"**，不在 boot 阶段就 spawn eva:sessions。
- 给 eva-sessions.mjs 加 `--format json` flag（10 行改动），用 stable JSON 作为 Claude 的契约面，ANSI 表格只给人眼。这条可以放进 Phase 0 deferred list（§8），但要在 §10 初始填充里登记为 `eva-sessions-json-output` P2 待办。

### F5. §3.6 "收线" 流程与 inbox / lesson-store 已有机制重叠 — [MAJOR]
**Where**: §3.6 收线流程 / §8 defer 列表
**Lens**: 企业管理系统贴谱性（事实=个人 Vessel，但 Phase 2 升级时的接口边界）
**Issue**:
- §8 显式说"完成自动写 lesson" defer。但 §3.6 步骤 3 "outcome 是 done/blocked/dropped" + 写 `completed_at + refs` 已经基本等同于 [memory/lesson-store.ts](../packages/backend/src/memory/lesson-store.ts) 里 LessonKind = 'review_closeout' 的内容。
- 同时 inbox-store.ts 已经存在 `triage: ideas | archive` 的 lifecycle（[inbox-store.ts:23](../packages/backend/src/inbox-store.ts)）。BACKLOG 的 `dropped` 是不是 `archive`？`blocked` 与 inbox 的 `open` 状态是什么关系？
- §3.5 / §3.6 流程是"BACKLOG-write → eva.json edit → commit"——纯文件操作，但 inbox / lesson 系统是 HTTP + DB。**Phase 0 把工作流绕开了 backend**，意味着 backend 后续提供的 inbox / lesson API 跟 BACKLOG 脱钩。

**Why it matters**:
- HARNESS_ARCHITECTURE.md §1 L5 列了"两个数据源"（harness.db + ~/.claude/projects）。再加一个 docs/BACKLOG.md 是合理的（人类编辑友好），但**与 inbox.jsonl + memory.db 的边界**需要 ADR 化。
- §6 I6 写"不复制 inbox 内容到 backlog ——引用即可（refs: [...]）"。但 inbox `id` 是 UUID，BACKLOG `refs` 怎么引用？没说格式。
- Phase 1+ 把 inbox 自动 promote 到 BACKLOG 时（§8 defer 项），需要现在就把"哪些字段从 inbox 继承"约定清楚——否则 schema 漂移。

**Suggested fix**:
- §3.1 schema `refs` 字段写明：`refs` 数组元素的格式是 `<kind>:<id>`，例如 `inbox:abc-123-uuid`、`pr:#42`、`commit:e8d2f81`、`lesson:<lesson-id>`。这是 contract，应该 lock。
- 在 §4 加 I7：BACKLOG 不直接写入 inbox / memory.db；反向也不许。两套系统**只通过 refs 字段**互相指认。
- §8 deferred 列表里"完成自动写 lesson"加注释："Phase 1 时按 BACKLOG.id 自动 import 进 memory.lessons 表，import_fingerprint 用 backlog id（lesson-store.ts 已有 unique index）"——这一行让 Phase 1 实施者知道接口已 ready。

### F6. 6 个 prompt 覆盖不够 80% — 至少缺 4 类高频操作 — [MAJOR]
**Where**: §3.2 6 prompt 表 / §2 G4
**Lens**: 里程碑裁剪（用户接口完整性）
**Issue**:
仔细看 §3.5/§3.6 流程后，实际频繁需要的操作明显不止 6 个：

| 操作 | 是否在 §3.2 | 用户视角必要性 |
|---|---|---|
| 看下一步 | ✓ | 高 |
| 开始一项 | ✓ | 高 |
| 收一项 | ✓ | 高 |
| 加一项 | ✓ | 高 |
| 标 blocked | ✓ | 高 |
| 看活窗口 | ✓ | 中 |
| **改优先级 / 改 priority** | ✗ | **高** — 现实中 P2 → P1 是常态 |
| **看 done 历史 / 列最近一周完成** | ✗ | **中-高** — 周回顾用 |
| **unblock 一项** | ✗ | **中** — blocked 解锁场景 |
| **dropped a item** | ✗ | **中** — 用户要"算了不做了" |
| 改 title / 改 note | ✗ | 低（直接改文件可接受） |
| 重命名 id | ✗ | 极低（I2 明令禁止） |
| 临时挂起 / pause | ✗ | 低（用 blocked 替代） |
| 拆分一项 | ✗ | 低（手工编辑） |

- §2 G4 说"6 个覆盖 80% 操作"。**至少 unblock + 改 priority + dropped 是高频**——加进去就 9 个，不是 6 个。
- "看活窗口"现实使用率低（一天 0-1 次）——可以并入"看下一步"的输出。腾出 slot 给更高频的 unblock。

**Why it matters**:
- 用户契约：用户被告诉"只要记 6 个短语"。如果第二周开始 4 类高频操作没短语，用户会 free-text，§6 R3 已经预见这个风险但只给软兜底"Claude 主动建议"。
- 6 个 prompt 是 contract（用户面）。Phase 1 加新短语需要更新 STEWARD_PROMPTS.md + 教用户重新记忆——成本不小。

**Suggested fix**:
- §3.2 调整成 8 个：去掉"看活窗口"（并入"看下一步"输出），加 3 个：
  - `<task-id> 改 P<n>` — 改 priority
  - `<task-id> 解锁` — 从 blocked 转 planned（auto-clear blocked_reason）
  - `<task-id> 不做了 <reason>` — drop
- 或者保留 6 个但把"加待办"做成"加/改"复合：`加待办 / 改 <task-id>: P<n>; ...` —— 单 prompt 多功能，认知负担依然 6。
- §3.2 加一个 fallback：用户 free-text "X 改成 P1"时 Claude 主动 normalize 到正式短语，确保意图归口。

### F7. CLAUDE.md boot ritual 在 Cursor 里"靠约定" — 可靠性证据不足 — [MAJOR]

### F9. §10 初始填充与 IMPROVEMENTS.md / IDEAS.md 实质重复 — [MINOR]
**Where**: §10 初始填充 / §1 problem 表
**Lens**: 企业管理系统贴谱性（个人单机但数据治理纪律）
**Issue**:
- §10 初始 active 项里 `m2-voice-proposal` 在 docs/IDEAS.md 里已经有占位（P1 / 中价值段）；`testflight-encryption-compliance` 也已经在用户记忆里。

### F10. boot ritual "Backlog stale" > 24h 阈值太短 — [MINOR]
