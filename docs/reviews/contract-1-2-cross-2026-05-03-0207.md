## Findings

### B1 [BLOCKER] migration 不是事务性的，失败后可能把 `user_version` 提前

**Where**: `packages/backend/src/migrations/0001_initial.sql` 的 `PRAGMA user_version = 100;` + `packages/backend/src/harness-store.ts` 的 `db.exec(sql)`  
**Lens**: 正确性 / 安全  
**Issue**: migration 文件一开始就设置 `user_version = 100`，但 `harness-store.ts` 没有包事务；如果后续 DDL 中途失败，DB 可能留下半成品 schema，同时下次启动因为版本已到 100 而跳过迁移。  
**Why this is a blocker**: 这是 schema 一旦 ship 后最难排查的损坏形态：表、触发器、FTS 可能缺一半，但版本号显示“已完成”。  
**Suggested fix**: 不要在 SQL 文件开头设置 `user_version`。在 `runMigrations` 里用 `db.transaction(() => { db.exec(sqlWithoutVersion); db.pragma("user_version = 100"); })`，并让每个 migration 明确声明目标版本，由 TS 代码在事务最后推进版本。

### M1 [MAJOR] `runMigrations` 未来会重跑所有旧 migration

**Where**: `packages/backend/src/harness-store.ts`，`if (current < HARNESS_SCHEMA_VERSION) { runMigrations(db, current); }`  
**Lens**: 正确性 / 不可逆  
**Issue**: 当前逻辑只判断全局版本小于目标版本，然后对 `migrations/` 下所有文件排序后全部执行，没有按当前版本跳过已应用 migration。  
**Suggested fix**: 建立 `0001 -> 100`、`0002 -> 101/200` 这类映射，只执行目标版本大于当前 `user_version` 的 migration；或者建 `schema_migrations` 表记录已应用文件。

### M2 [MAJOR] `stage(issue_id, kind)` 应该是业务唯一，但现在只是普通索引

**Where**: `packages/backend/src/migrations/0001_initial.sql`，`CREATE INDEX idx_stage_issue_kind ON stage(issue_id, kind);`  
**Lens**: 正确性 / 不可逆  
**Issue**: 文档说每个 Issue 创建固定 10 个 Stage，且 `kind` 固定顺序；但 DB 允许同一个 issue 下出现多个 `spec` / `implement` stage。  
**Suggested fix**: 改成 `CREATE UNIQUE INDEX ... ON stage(issue_id, kind)`。如果未来需要重跑某个 stage，用 `attempt` / `revision` 字段表达，不要让主 stage 节点重复。

### M3 [MAJOR] Artifact 的 dedupe 语义和 `stage_id` 关系冲突

**Where**: `docs/HARNESS_DATA_MODEL.md §1.10` 的“相同内容自动去重（先查 hash 命中即复用 row）” + `artifact.stage_id NOT NULL`  
**Lens**: 正确性 / 跨端对齐  
**Issue**: 如果相同 hash 复用同一 row，这个 row 只能属于一个 `stage_id`；如果不复用 row，`idx_artifact_hash` 又只是普通索引，不能保证 dedupe。  
**Suggested fix**: 二选一：  
- Artifact row 表示一次 stage 产物：保留 `stage_id`，把文档里的“复用 row”改成“复用 file content，但每个 stage 一行”。  
- Artifact row 表示全局内容：移除 `stage_id`，新增 `stage_artifact(stage_id, artifact_id, kind)` 关联表。

### M4 [MAJOR] `superseded_by` 链方向写反了

**Where**: `docs/HARNESS_DATA_MODEL.md §1.10`，`superseded_by TEXT REFERENCES artifact(id)` + “新 row + superseded_by 链回旧 row”  
**Lens**: 正确性 / 不可逆  
**Issue**: 字段名 `superseded_by` 的自然含义是“当前 artifact 被谁替代”；但文档说新 row 指回旧 row，这会表达成“新内容被旧内容替代”。  
**Suggested fix**: 要么旧 row 更新 `superseded_by = new_id`；要么字段改名为 `supersedes_id`，由新 row 指向旧 row。

### M5 [MAJOR] `issue.retrospective_id` 文档有 FK，migration 没有 FK

**Where**: `docs/HARNESS_DATA_MODEL.md §1.3` vs `packages/backend/src/migrations/0001_initial.sql` 的 `retrospective_id TEXT`  
**Lens**: 跨端对齐 / 正确性  
**Issue**: 文档定义 `retrospective_id TEXT REFERENCES retrospective(id)`，实际 SQL 去掉了 FK，只留下注释说“避免环依赖”。SQLite 可以创建引用后声明的表，这里不需要牺牲约束。  
**Suggested fix**: 恢复 `retrospective_id TEXT REFERENCES retrospective(id)`；如果担心环依赖插入顺序，使用可空字段和应用层更新即可。

### M6 [MAJOR] schema 验收脚本的 Artifact CHECK 测试会因为别的 FK 先失败

**Where**: `packages/backend/src/test-harness-schema.ts`，先插入 `stage(... methodology_id='m1')`，后插入 `methodology('m1')`  
**Lens**: 正确性  
**Issue**: 这段 try/catch 期望验证 `artifact CHECK rejects inline without content_text`，但实际会先因为 `stage.methodology_id` FK 不存在而抛错，测试结论是假阳性。  
**Suggested fix**: 先插入 `methodology`，再插入 `stage`，最后单独插入非法 artifact；并断言错误信息包含 `CHECK constraint failed`。

### M7 [MAJOR] audit log 的“所有写操作必有审计”没有失败语义

**Where**: `docs/HARNESS_DATA_MODEL.md §1.14` + `docs/adr/ADR-0010-sqlite-fts5.md`  
**Lens**: 安全 / 运维风险  
**Issue**: 文档要求所有 INSERT / UPDATE / DELETE 同步 append audit JSONL，但 audit 在 SQLite 事务外。DB 写成功、JSONL append 失败时，系统会出现不可审计写入。  
**Suggested fix**: 明确语义：如果 audit 是强一致要求，把 audit entry 放进 SQLite 同事务；如果坚持 JSONL，必须定义“DB commit 前预写 pending audit / commit 后 finalize”的恢复流程，或者把 audit 降级声明为 best-effort。

### m1 [MINOR] boolean-like 字段没有 CHECK 约束

**Lens**: 正确性

**Suggested fix**: 加 `CHECK (field IN (0,1))`；`agrees_with_prior` 保持 `NULL OR IN (0,1)`。
