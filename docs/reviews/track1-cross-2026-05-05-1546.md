### B1 [BLOCKER] agentProfiles fixture change breaks the existing drift test

**Where**: `packages/shared/src/__tests__/m0-agent-profiles.test.ts:72`, `packages/shared/fixtures/harness/fallback-config.json:76`

**Lens**: 正确性 / 跨端对齐

**Issue**: fallback config now enables `Strategist`, `PM`, and `Coder`, but the existing test still asserts exactly one enabled profile and that it must be `PM`.

**Why this is a blocker**: This should fail the shared test suite before merge. The failure is not cosmetic: the test encoded the old M0 contract ("PM is the only enabled profile in M0"), while Track 1 intentionally changes the M1 runtime contract. The test must move with the contract.

**Suggested fix**: Update the test name and assertions to the new M1 invariant, for example enabled IDs equal `["Coder", "PM", "Strategist"]`, with stages `implement`, `discovery`, and `strategy`. If PM is no longer used by scheduler but remains enabled for discovery/UI, say that explicitly in the test name.

### B2 [BLOCKER] The failed-stage recovery instruction does not make the issue schedulable again

**Where**: `packages/backend/src/scheduler.ts:60`, `packages/backend/src/scheduler.ts:67`, `packages/backend/src/scheduler.ts:47`

**Lens**: 正确性 / 不可逆

**Issue**: When any stage is `failed`, `tick()` updates the issue to `blocked` and returns the instruction "delete failed stage row(s) and re-tick". But `ELIGIBLE` only includes `triaged`, `planned`, and `in_progress`; once the issue is `blocked`, deleting the failed stage row is not enough. The next tick will skip it.

**Why this is a blocker**: The new behavior is specifically a manual recovery path. As written, following the recovery text leaves the issue stuck unless the operator independently knows to change `issue.status` back to an eligible value. This is the kind of state-machine trap that caused the original UNIQUE defect.

**Suggested fix**: Either keep the issue status eligible and return a blocked reason without writing `blocked`, or make the resolution text and implementation explicit: delete/resolve the failed stage and set the issue back to `planned` or `in_progress` before re-tick. If the intended M1 path is direct SQLite surgery, include the exact minimal SQL or add a small backend route for "resolve failed stage for retry".

### M1 [MAJOR] The blocked recovery path points to a delete operation that the current API does not expose

**Where**: `packages/backend/src/routes/harness.ts` stage routes, `packages/backend/src/scheduler.ts:67`

**Lens**: 正确性 / 简化

**Issue**: The reason string says "deleting failed stage row(s)", but the harness route surface has list/create/update status for stages and no `DELETE /stages/:id`. So the documented recovery path is not available from the product API.

**Why this matters**: M1 can accept manual operator steps, but the current wording reads like a normal product action. In practice, recovery requires direct DB access or an unimplemented route.

**Suggested fix**: For M1, either change the text to "manual DB cleanup" and document the exact SQL, or add the narrowest safe operation: mark failed stage as skipped/rejected plus reset issue status, or delete failed stage behind an explicit admin endpoint.

### M2 [MAJOR] iOS settings copy is now stale after enabling multiple profiles

**Where**: `packages/ios-native/Sources/ClaudeWeb/Views/Settings/SettingsView.swift:252`, `packages/shared/fixtures/harness/fallback-config.json:76`

**Lens**: 跨端对齐

**Issue**: The iOS Settings footer still says "M0 期仅 PM 启用", but the server-driven config now enables `Strategist` and `Coder` too.

**Why this matters**: The UI will render the new enabled chips from server config, while the footer says only PM is enabled. That is a visible contradiction in the same screen.

**Suggested fix**: Update the footer to describe the M1 state in neutral terms, e.g. "当前启用的 profile 由服务端配置决定；M1 scheduler 使用 strategy/implement 对应 profile。"

### M3 [MAJOR] Coder uses `modelHint: "adaptive"` but scheduler silently maps it to Sonnet

**Where**: `packages/shared/fixtures/harness/fallback-config.json:122`, `packages/backend/src/scheduler.ts:183`

**Lens**: 跨端对齐 / 简化

**Issue**: `Coder` is now enabled with `modelHint: "adaptive"`, but `spawnAgent()` only branches on `opus` and `haiku`; every other value becomes `claude-sonnet-4-6` silently.

**Why this matters**: `adaptive` is valid as a hint-only protocol field, and iOS even displays it as a distinct orange chip. But scheduler runtime treats it as Sonnet with no comment or warning. That makes the UI/config contract look richer than the actual execution contract.

**Suggested fix**: For M1, either set Coder to `modelHint: "sonnet"` until adaptive routing exists, or add an explicit `adaptive` branch/comment that intentionally resolves to Sonnet in M1. Avoid silent fallthrough.
