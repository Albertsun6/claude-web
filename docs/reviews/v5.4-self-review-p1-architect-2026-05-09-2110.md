## BLOCKER（3 条）

### B-A1: Coding Driver 与 5 接口契约的归属关系不清

**位置**：plan §「关键依赖」M0.5 段；§「目录结构」`packages/backend/src/drivers/coding/`；§「关键设计决策」未提；M0.5 段提到 `submit_coding_task` 接口契约。

**问题**：v5.4 plan 多次提"把 cli-runner.ts 按 Vessel `submit_coding_task` 接口契约重构"，但 plan §「0A 完成判定」写 "FRAMEWORK 5 接口签名"——5 接口 = Agent/Skill/Tool/Memory/App。**Driver 不在这 5 接口内**。所以 0A FRAMEWORK 落地时会出现：
- 路径 A：扩到 6 接口（Agent/Skill/Tool/Memory/App + **Driver**），与 ARCHITECTURE.md / CONCEPTS.md "5 接口"不一致
- 路径 B：保持 5 接口，CodingDriver 是 Tool 接口的实现 —— 但 Tool 是 MCP 协议级别的工具，CodingDriver（spawn CC CLI）颗粒度不同

**架构后果**：0A FRAMEWORK 写接口签名时这两条路径会撕裂；后续 M0.5 / M1B 实现互相干扰。

**建议**：0A 之前在 plan 里显式声明走 A 还是 B，并更新 ARCHITECTURE / CONCEPTS 一致。或者在 ADR-014 增订段 / 新加 ADR-016 锁定。

### B-A2: Boot 三层（CONCEPTS §3.5）在 Eva backend/src/index.ts 上的落地路径未规划

**位置**：plan §「关键依赖」M0；§「目录结构」`packages/backend/src/index.ts # 进程入口（沿用 Eva）`；EVA_TO_VESSEL_MAPPING 没列 index.ts 的重构动作。

**问题**：CONCEPTS §3.5 三层 boot（进程级 / Instance / Session）是 v0.3 架构的核心。Eva backend/src/index.ts（从盘点知 L64 起 Hono app / L250+ serve / L271+ 挂载 harness router / L294+ /ws upgrade）是单层 boot。Vessel 三层 boot 必须改造 index.ts，但 plan 0-pre / 0A / M0 都没说"index.ts 怎么从单层改三层"。

**架构后果**：
- M2-Soul 时"读 soul.md 注入 system prompt"对应 §3.5.2 Instance 级初始化——如果 index.ts 还是单层，soul.md 改动后必须重启整个 vessel-core（违反 §3.5 "三层独立可重入"原则）
- M1A session_id 共享对应 §3.5.3 Session 级——同样依赖三层划分

**建议**：在 EVA_TO_VESSEL_MAPPING 里新增一行 `index.ts → boot.ts`，重构动作 = "拆三层 boot 函数"，落到 M0。

### B-A3: 5 接口的存放位置与 Wire Protocol 的关系混乱

**位置**：plan §「目录结构」`packages/backend/src/interfaces/{agent,skill,tool,memory,app}.ts`；同时 `packages/shared/src/protocol.ts` 含 ClientMessage/ServerMessage（Wire Protocol）。

**问题**：5 接口（运行时契约）放 backend/，Wire Protocol（HTTP/WS 通信协议）放 shared/。但 0A FRAMEWORK 写"Intent / Artifact / Session 字段表（基于 Eva shared/protocol.ts 扩展）"——Intent 是不是 5 接口的输入？如果是，则 Intent 需要在 shared/ 让 frontend / iOS import，但运行时接口契约又在 backend/，两个地方需要同步演进。

**架构后果**：5 接口可能依赖 shared/ 的 Intent 类型 → 5 接口被迫也放 shared/；但 5 接口里 Memory / App 是后端纯逻辑，frontend 不需要 → 又要拆。

**建议**：0A FRAMEWORK 锁定：① Intent / Artifact / Session 这种**跨端数据类型**放 `shared/`；② 5 接口（Agent/Skill/Tool/Memory/App，纯后端运行时契约）放 `backend/src/interfaces/`；③ 5 接口可以 import shared/ 的数据类型（单向依赖）。这条规则需要在 ADR-010 docs-organization 或 ADR-001 package-manager 增订段写明。

## MAJOR（4 条）

### M-A1: M2-Soul prompt wrapper 注入的"长期演进副作用"未在文档落实

**位置**：ADR-004（v5.1 评审者认可）；plan §「关键设计决策 / Soul Spec 注入目标」。

**问题**：ADR-004 锁定 Soul 注入到 cli-runner prompt wrapper —— 这意味着**未来非 cli-runner 的 Skill**（如 M1B 直接调 MCP server，或 M2-Voice 走 Voice Capability + ML worker，不经 cli-runner）**不带 soul prompt**。第五轮评审者认可这个 trade-off，但 plan 只在 ADR-004 Consequences 提了一句，**ARCHITECTURE.md / CONCEPTS.md 没有同步**。

**架构后果**：未来用户报"EVA 在某些场景没人格"，定位时会找不到原始决策依据；新加 Capability 的开发者不知道这个边界。

**建议**：CONCEPTS §1.2（Soul Spec / Instance 章节）补一段"Soul Spec 作用范围"，明确写 "v0.1 仅作用于走 CodingSkill 的 Intent；v1+ 可扩展到所有 Skill"。

### M-A2: ML worker 在 5 接口里的归属未明

**位置**：ADR-012；目录结构 `ml-workers/` 顶层；§「关键设计决策 / ML Worker 边界」。

**问题**：ML worker 是后端调用（embedding / ASR / TTS），从 architect lens 看：
- 它是 **Memory 接口**的实现细节（向量索引 = embedding worker）？
- 还是 **Tool 接口**的实现（embedding-as-a-tool）？
- 还是单独的 **MLDriver**（5 接口外）？

plan 没说。`packages/backend/src/ml-worker/` 目录下 manager.ts + 3 个 client.ts，都是裸 TS 模块，没说实现哪个接口。

**建议**：0A FRAMEWORK 锁定：每个 ML worker client 实现哪个 Vessel 接口（推荐：embedding-client 是 Memory 内部 helper，asr-client / tts-client 是 Voice Capability 的 helper）。

### M-A3: telemetry-store.ts 与 observability/trace.ts 共存策略未定

**位置**：EVA_TO_VESSEL_MAPPING 标"telemetry-store.ts → Trace 协议 / 按 OpenTelemetry-lite 12 字段重构"；目录结构 `observability/trace.ts`。

**问题**：是 in-place 改 `telemetry-store.ts`，还是新建 `observability/trace.ts` 取代它？如果两者并存，trace_id 字段在 telemetry 表 vs trace event 文件里会双写。

**建议**：0A 设计阶段或 0-pre 时定方向 —— 推荐 in-place 改 `telemetry-store.ts` 为 12 字段（Eva 现有调用方迁移成本低），同时把 `observability/trace.ts` 定位为"client 接口 + 子进程环境变量传播协议"。EVA_TO_VESSEL_MAPPING 加一行明确动作。

### M-A4: Workflow / harness 在 5 接口外的位置不一致

**位置**：plan §「目录结构」`packages/backend/src/workflow.ts`；EVA_TO_VESSEL_MAPPING `scheduler.ts → Workflow Engine`；ARCHITECTURE.md §3 控制平面。

**问题**：Workflow 在 ARCHITECTURE 里属于"控制平面"（与 Orchestrator / HITL Gate / Session 同层），不在 5 接口里。但 plan 没说 Workflow 是不是控制平面的子模块。如果是，目录应该 `control_plane/workflow.ts` 而非 `workflow.ts` 平级——保持与 ARCHITECTURE 5 层一致。

**建议**：目录结构调整：`packages/backend/src/control_plane/{orchestrator.ts, workflow.ts, session.ts, hitl.ts}` —— 跟 ARCHITECTURE §3 的 5 层架构一一对应。

### m-A2: 5 接口（Agent/Skill/Tool/Memory/App）顺序不一致

**位置**：多处。plan 0A 完成判定第 2 条写 "Agent, Skill, Tool, Memory, App"；目录树写 `{agent,skill,tool,memory,app}.ts`；CONCEPTS §1.4 八件套辨析有不同顺序。

**问题**：顺序不一致只是文档质量问题，但代码里 `import { Agent, Skill, Tool, Memory, App }` 跟 `import { App, Memory, Tool, Skill, Agent }` 出现两种风格会让 grep / replace 出错。

**建议**：0A FRAMEWORK 锁定一个 canonical 顺序（推荐按依赖：App → Skill → Agent → Tool → Memory，或按字母：Agent / App / Memory / Skill / Tool）。
