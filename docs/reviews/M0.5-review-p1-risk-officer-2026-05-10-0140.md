## BLOCKER

### R-M0.5-1：NFR-F1 5s 预算与 cli-runner SIGKILL 共时 → 孤儿进程

**位置**：
- `packages/backend/src/cli/vessel-core.ts:90-93`：vessel-core hard-exit timer = `setTimeout(() => process.exit(...), 5000)`
- `packages/backend/src/cli-runner.ts:104-114`：cli-runner SIGKILL escalation timer = `setTimeout(SIGKILL, KILL_GRACE_MS=5000)`

**Failure mode**：

```
T = 0   ：用户按 Ctrl-C → vessel-core onShutdown('SIGINT')
        → abortCtl.abort()                      // 触发 cli-runner-driver onParentAbort
        → setTimeout(hardExit, 5000).unref()    // V 计时器 V@5s
T = 0+ε ：cli-runner-driver onParentAbort → driver-内 abortCtl.abort()
        → cli-runner.onAbort 触发 → process.kill(-pid, SIGTERM)
        → setTimeout(SIGKILL, 5000)             // C 计时器 C@5s
T = 5000：V 与 C 几乎同时 fire；JavaScript event loop 调度顺序取决于内部
        macrotask 排队，**两者都在同一 tick 上**。
        如果 V 先 fire → process.exit(130) → C 计时器随父进程死掉
        → 子进程 group 仅收到过 SIGTERM，detached → 孤立运行
```

**Detached + SIGTERM-only + parent exit** 是 **真实的孤儿场景**：

- `detached: true` 意味着 child group 不绑父进程生命周期
- claude CLI 子进程若卡在 hook fetch / network IO，5s 内没响应 SIGTERM
- 父 vessel-core 进程在 5s 整点 `process.exit()`，C 计时器（在父进程的 timer queue 里）随之消失
- 子 group **持续运行**，可能继续写 `<DATA_DIR>/workspace/<runId>/`，**绕过取消语义**

这触及 4 类硬触发 #8（数据破坏）的弱形态：用户主动取消后，磁盘上仍可能出现"取消后写入"的文件，违反 NFR-F1 cancellation contract。

**修复（必需）**：

让 cli-runner 的 SIGKILL **早于** vessel-core hard-exit。两种简单做法：

```typescript
// 1. 缩短 cli-runner KILL_GRACE_MS
const KILL_GRACE_MS = 4000;   // ↓ 4s

// 2. 或拉长 vessel-core hard-exit
setTimeout(() => process.exit(...), 6000).unref();   // ↑ 6s
```

并在 RISKS.md 增订一条 **R-NFR-F1-budget**：vessel-core 的"5s 预算"实质应 ≥ cli-runner SIGKILL escalation + reaper margin（建议 +1s）。

也可以在 vessel-core hard-exit 之前先 `process.kill(-childPid, SIGKILL)` 一次（但需要从 cli-runner 暴露 pid，改动较大）；前述错峰方案改动小、修得彻底。

## MAJOR

### R-M0.5-2：cli-runner 不处理 already-aborted signal — 早期取消失效

**位置**：`packages/backend/src/cli-runner.ts:116`

```typescript
p.signal?.addEventListener("abort", onAbort);
```

**问题**：`AbortSignal.addEventListener('abort', ...)` 仅监听**未来**的 abort 事件。如果 `p.signal.aborted === true` 在 addEventListener 调用之前已经成立，listener 永远不会触发，spawn 出来的 child 不被杀。

**触发路径**：

cli-runner-driver.ts:88-90 写得正确：
```typescript
if (ctx.abortSignal.aborted) abortCtl.abort();
else ctx.abortSignal.addEventListener('abort', onParentAbort);
```

但 `abortCtl` 被同步 abort 后，下一行 `runSession({signal: abortCtl.signal, ...})` 进 cli-runner runOnce → 立即 `spawn(...)` → 然后 line 116 `addEventListener` —— 此时 signal 已 aborted，listener 永不 fire。**child 已经在跑**，没人给它发 SIGTERM。

虽然 line 173 `signaled: ... || !!p.signal?.aborted` 让上层认为 "signaled = true"，但**子进程实际上没死**——只是 close handler 之后 socket 断了，CC CLI 还在跑（它写工作区、写网络）。

**修复（必需）**：在 cli-runner.ts:116 后立刻补一次 already-aborted 检查：

```typescript
p.signal?.addEventListener("abort", onAbort);
if (p.signal?.aborted) onAbort();    // ← 防早期取消逃逸
```

测试：在 test-coding-driver.ts 增订 "abort BEFORE submit returns spawn" 的 race test：

```typescript
const abortCtl = new AbortController();
abortCtl.abort();   // 先 abort
const result = await runIntent({ codingDriver: realDriver, abortSignal: abortCtl.signal });
// expect: 子进程已被 SIGTERM；exitCode 130；workspace 内无 partial write
```

### R-M0.5-3：`.input` 对象递归后非黑名单 leaf field 不被 path-redact

**位置**：`packages/backend/src/observability/trace-redactor.ts:114-136`

`redactValue` 当前实现：
- path 命中 SUFFIX **且** value 是 leaf（string/number/null）→ `***`
- path 命中 SUFFIX **且** value 是 object/array → **递归**，依赖子 leaf 自己再命中 SUFFIX 或正则

CC stream-json tool_use 的典型 payload：

```json
{
  "type": "tool_use",
  "input": { "command": "ls /Users/yongqian/private", "description": "list" }
}
```

走 redactValue：
- path = `payload.message.content[0].input` → SUFFIX 命中 `.input`，但 value 是 object → 递归（不 mask 整体）
- path = `payload.message.content[0].input.command` → SUFFIX 命中 `.command` → leaf string `"ls /Users..."` → `***` ✓
- path = `payload.message.content[0].input.description` → SUFFIX 命中 `.description` → leaf string → `***` ✓

但如果 CC 把工具参数写到一个**未列入 SUFFIX 的字段名**（比如 `input.args`、`input.payload`、`input.body`、`input.query`），则：

- path = `payload.message.content[0].input.args` → SUFFIX **不命中**
- value 是 string → 走 `redactString` 正则兜底
- 兜底正则只抓 sk-/AKIA-/email/path/20+ char token —— 短工具参数（比如 user prompt 的 30 char 摘要 `"帮我读一下 ~/Documents/x"`）**漏掉路径**虽然命中 `/Users/[name]/` 正则——但更短/更随意的 prompt 片段不一定命中。

**Risk**：未来 CC 升级 / Cursor 接入时，新字段名会绕过 SUFFIX 列表，**敏感工具参数**写到 trace 文件。

**修复（建议）**：在 SUFFIX 检查后增订 **"SUFFIX 命中 object 时递归遮蔽所有 leaf"** 的硬规则：

```typescript
function redactValue(value: unknown, path: string): unknown {
  const pathHit = isPathRedacted(path);
  if (pathHit && (value === null || typeof value !== 'object')) return '***';
  if (typeof value === 'string') {
    return pathHit ? '***' : redactString(value);
  }
  if (Array.isArray(value)) {
    return value.map((v, i) =>
      pathHit
        ? redactValueForceMask(v, `${path}[${i}]`)   // ← path 已命中，子树内全部强制 mask leaf
        : redactValue(v, `${path}[${i}]`));
  }
  if (value && typeof value === 'object') {
    const out: Record<string, unknown> = {};
    for (const [k, v] of Object.entries(value)) {
      out[k] = pathHit
        ? redactValueForceMask(v, `${path}.${k}`)
        : redactValue(v, `${path}.${k}`);
    }
    return out;
  }
  return value;
}

function redactValueForceMask(value: unknown, path: string): unknown {
  if (value === null || typeof value !== 'object') return '***';
  // 递归，保留结构骨架
  if (Array.isArray(value)) return value.map((v, i) => redactValueForceMask(v, `${path}[${i}]`));
  const out: Record<string, unknown> = {};
  for (const [k, v] of Object.entries(value)) out[k] = redactValueForceMask(v, `${path}.${k}`);
  return out;
}
```

效果：`.input` / `.content` / `.message`（object）等"父字段一旦在黑名单"，整子树所有 leaf 强制 `***`，**结构 schema 仍保留**用于调试。

测试：增订 `tests/redactor.test.ts` —— 喂 fixture：

```typescript
{
  message: { content: [{ type: 'tool_use', input: { args: 'rm -rf', custom_field: 'secret' } }] }
}
```

期望 `args` / `custom_field` 都是 `***`。

### R-M0.5-4：`payload.message` string vs object 行为差异

**位置**：trace-redactor.ts:119

当前正确：
- `payload.message` = string（如 system event `{message: "stale_session_recovered"}`）→ `.message` SUFFIX 命中、leaf → `***` ✓
- `payload.message` = object（CC stream-json `message: {role, content: [...]}`）→ SUFFIX 命中但 typeof === 'object' → 递归 ✓

**潜在缺陷**：递归后依赖**所有** leaf 都命中 SUFFIX 才算干净。同 R-M0.5-3，未来 CC schema 加新字段，可能裸串泄露。

**修复**：和 R-M0.5-3 同一 fix（`forceMask` 模式）。

**额外建议**：在 redactor.ts 顶部加一段 invariant 注释：

```typescript
// INVARIANT: 一旦 path SUFFIX 命中，整个子树视作"敏感容器"——
// 即使新字段未列入黑名单，也要 mask leaf。否则 CC schema 演化会绕过。
```

## MINOR

### R-M0.5-5：`.summary` 在 SUFFIX 黑名单 — spillover 自吞

**位置**：trace-writer.ts:56 + trace-redactor.ts:51（`.summary` SUFFIX）

trace-writer 的 §5 spillover 自己 fabricate 一个 `{summary: payloadSummary(redactedPayload)}`，然后调 `TraceEventSchema.parse` 写盘。**当前**这个 summary 不会被再次 redactPayload，所以没事。

但如果未来重构把 redactPayload 移到 schema-validate 之后（防御深度——校验后再脱敏），`.summary` SUFFIX 会把 spillover 的 200-char 摘要整个 mask 成 `***`，调试能力清零。

**修复**：要么去掉 SUFFIX 里的 `.summary`（risk-officer 觉得 summary 字段不够"通用敏感"以加入硬黑名单——CC schema 没用 `.summary` 表示 prompt 内容）；要么把 spillover 的字段名改成 `.redacted_summary`（不在 SUFFIX 里）以避开。

### R-M0.5-6：mkdir 0o700 在已存在目录不 enforce mode

**位置**：cli-runner-driver.ts:64-67、:95；trace-writer.ts:44；fake-coding-driver.ts:47

`mkdirSync(dir, {recursive: true, mode: 0o700})` 仅在**实际创建**目录时应用 mode。已存在的目录保持原 mode。

**Failure mode**：
- 用户从某个 0.x dev build 升级，`<DATA_DIR>/workspace` 留下 0o755（pre-M0.5 默认）
- 升级到 M0.5 后 mkdirSync 不抛错（recursive 容忍 EEXIST），但权限**仍是 0o755**
- 同主机上其他用户（如果是 shared mac）能 ls / read 工作区目录树

**修复**：在 mkdirSync 之后 `chmodSync(dir, 0o700)` 强制刷一次：

```typescript
if (!existsSync(workspaceRoot)) {
  mkdirSync(workspaceRoot, { recursive: true, mode: 0o700 });
} else {
  // Defense: ensure mode if dir pre-existed from older build
  chmodSync(workspaceRoot, 0o700);
}
```

同样适用于 `expected` (`<workspace>/<runId>`) 和 `traceDir`。**优先级低**，因为 Vessel 是个人单机，但 ADR-014 4 类硬触发 #5 secrets 提"文件权限"——加固有用。

### R-M0.5-7：DATA_DIR 自身是 symlink 时 realpath-startsWith 不能识别

**位置**：cli-runner-driver.ts:69-73

`realpathSync(workspaceRoot)` resolves 符号链接 → 拿到真实路径。但**真实路径本身可能是攻击者指定的目录**（例如 `~/.vessel` 是个 symlink 到 `/tmp/foo`）。`startsWith` 检查只能验证 child 在 root 下，不能验证 root 是否合法。

**Realistic threat**：低——这要求攻击者已经能写 `~/.vessel/` 自身或 `~/`。如果到了这一步，整个 vessel 数据已经被劫持。

**Mitigation**：在 startup 时（vessel-core.ts main）一次性校验 DATA_DIR 不是 symlink + 在 ~ 下：

```typescript
import { lstatSync } from 'node:fs';
const lst = lstatSync(DATA_DIR);
if (lst.isSymbolicLink()) throw new Error('VESSEL_DATA_DIR must not be a symlink');
```

**优先级低**，但是 cheap 加固。

### R-M0.5-8：5s 预算"边界"语义

**位置**：cli/vessel-core.ts:88 vs cli-runner.ts:34

NFR-F1 文本（ROADMAP）：「SIGINT 后 **5 秒内**进程退出」。当前实现两个 5s 计时器**共时**触发，实质 budget = 5s（理想情况）或 5s + tiny（race）。

**问题**：vessel-core hard-exit 在 5s 整点；如果 cli-runner 的 SIGKILL 还没来得及把 child reaped，子进程被孤立（见 BLOCKER R-M0.5-1）。即使错峰修了，文档应明确：

- **5s 预算指**：SIGINT → vessel-core 父进程 exit
- **不保证**：5s 后 detached child 已死

或者干脆：将 vessel-core hard-exit 抬到 6s（child SIGKILL @ 5s + 1s margin）。

### R-M0.5-9：unparseable line 完全无记录

**位置**：cli-runner-driver.ts:135-137

```typescript
catch {
  return JSON.stringify({ unparseable_line_redacted: true });
}
```

unparseable line 的**长度 / hash6 / 时间**都没记录。后续 debug 时无法判断"丢了多少行 / 什么时候发生"。

**Mitigation**：

```typescript
catch {
  return JSON.stringify({
    unparseable_line_redacted: true,
    line_len: line.length,
    line_hash6: hash6(line),
  });
}
```

**优先级低**（仅 debugging 增强）。

### R-M0.5-10：跨平台 stat 校验脚本缺失

**位置**：trace-redaction-spec.md §7 acceptance C-1/C-2

spec 写了 macOS `stat -f` 和 Linux `stat -c`，但没在 acceptance 里给统一脚本。0A FRAMEWORK / M0.5 closeout verify-gate 跑的话，CI/Linux box 上可能跑挂。

**Mitigation**：在 `scripts/check-trace-modes.sh` 加 platform detection：

```bash
case "$(uname)" in
  Darwin) STAT_FMT='-f %Lp';;
  Linux)  STAT_FMT='-c %a';;
esac
test "$(stat $STAT_FMT instance/traces/<trace_id>)" = "700" || exit 1
```

## 总评

**M0 closeout 修过的 redactor 主体方向 ok**——SUFFIX 列表 + 路径白名单 + 4KB spillover + 文件 0600 都到位，不是装样子。**但** 这次新增的 stdout 路径（driver 直接写 `<traceDir>/<span_id>.stdout`）+ 进程组取消语义把"5s 预算"和"redact 黑名单 SUFFIX 完整性"两个老问题暴露出来。

**两条必须修**（BLOCKER + 关键 MAJOR）：
1. **5s 预算与 SIGKILL 共时** → 错峰
2. **cli-runner 不处理 already-aborted signal** → 加 1 行 guard

**两条强烈建议修**（MAJOR，关乎未来 CC schema 演化）：
3. redactor SUFFIX 命中的 object 子树 forceMask
4. 测试覆盖 race + object-subtree fixture

其他 MINOR 在 v0.1 release 前清理即可。

不阻塞 M0.5 ship——**但 R-M0.5-1 / R-M0.5-2 必须在 M0.5 PR 合并前修**。
