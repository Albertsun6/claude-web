## 四维评审

### 架构可行性

13 实体 DTO + AuditLogEntry + HarnessEvent + 版本常量这套结构是合格的最小协议骨架：

1. **enum 锁实施到位**。HARNESS_PROTOCOL.md §1 明列每个 enum 的全部值；harness-protocol.ts 用 `z.enum([...])` 一一对齐；HarnessProtocol.swift 用带显式 raw value 的 enum 把 `in_progress`/`awaiting_review`/`claude-web` 等带连字符 / 下划线的值都做了正确映射；0001_initial.sql 的 CHECK 与之对齐。这条比预期做得更彻底。
2. **wire/DB 命名漂移有清晰约定**。HARNESS_PROTOCOL.md §1 + harness-protocol.ts 文件头都明文写 "DB snake_case，wire camelCase，转换由 store 层负责，不在本文"。这是契约对，但 store 层的转换器 ([packages/backend/src/harness-store.ts](../packages/backend/src/harness-store.ts)) 是契约 #1 范围、本文不审。
3. **HarnessEvent 用 Zod discriminatedUnion + Swift 手写 init/encode**：两端实现都正确处理 7 个 kind。但 Swift 端的手写 enum 没有抗"未知 kind"的 graceful 通道——后端推 v1.1 加 kind 时，老 iOS 立即 throw decoding error 而不是 fallback。详见 MAJOR-3。
4. **provisional persistence-only 注释（Stage.{input,output,review_verdict}ArtifactIds）**：harness-protocol.ts 和 Swift 文件都在源码里加了"M2 dogfood 反查询频率信号决定是否升级"的注释。这种"我们知道这块以后会改 + 先这么写不承诺向后兼容"的诚实标注，是契约 #2 比 #1 做得更专业的地方，**强烈保留**。
5. **AnyCodable 实现**：Swift 端的 AnyCodable 走 `singleValueContainer + try? decode 序列尝试 Bool/Int64/Double/String/Array/Dict`。这种实现常见的坑是 Int 被 Double 优先吃掉（导致 round-trip 后 `1` 变 `1.0`），但 Swift 这里 `Int64` 在 `Double` 前面尝试，对纯整数 JSON 字面量会先命中 Int64，**通过这关**。但注意 `Int64.decode` 对 `1.0`（带小数点的 JSON 数）是会失败的，所以实际行为是"正确的整数走 Int64，分数走 Double"——OK。

**最脆弱假设**：协议层把 `metadata: z.record(z.unknown())` / Swift 的 `[String: AnyCodable]` 当作长期占位，但 `metadata` 是 spec / design Artifact 的"企业字段必填段"载体（fixture artifact-inline.json 已经塞了 `businessEntities, permissionMatrix, approvalSteps, reportSchemas`）。M2 dogfood 把它升 typed schema 时是 minor bump 还是 major bump，目前 ADR-0015 没回答。

### 里程碑裁剪

M-1 把"协议层"切成"DTO + 版本常量 + 骨架 WS event + ADR placeholder"是合理的。BLOCKER 已说过 §8 与现实的不一致；除此之外里程碑切片本身没问题：

1. **enum 锁段**已显式列出 UI-driving enums（Stage.kind / Stage.status / Stage.weight / Issue.status / Task.model / Methodology.appliesTo），满足 Round 2 arch 的"必须显式列出"要求。
2. **fixture 全覆盖** 16 个 JSON 文件覆盖 13 实体 + audit + event + Artifact 的 inline/file 双形态。fixtures/README.md 有清单 + 对应 schema。这一项以前评审的 "fixture 必须能跑 round-trip" 已经达到。
3. **M0 vs M1 vs M-1 边界** 在 §0 和 §8 末尾两次明示——`/api/harness/config`、SchemaRenderer、ADR-0011 升 Accepted、`/api/harness/*` REST 端点都被推到对应里程碑。这条做得正确。
4. **WS event payload "字段名锁住，结构留给 M0 调整"**（§4）——这一条的"锁住"和 Swift / TS 当前实现是一致的；M0 时如果加字段是 minor bump，删字段是 major bump（按 ADR-0015 的规则）。但 Swift discrimination 没有 "未知 kind" 兜底（详见 MAJOR-3），意味着 M0 加新 kind 时**事实上是 major bump**——这条应该在 §4 末尾写明。

**最脆弱假设**：M-1 验收时 Swift 端只有"人工抽样验证 round-trip"（test 文件 line 4-5 注释）。这意味着 fixtures Swift round-trip 实际上没人跑过；而 audit-log-entry 的 `before: null` 在 Swift 默认 Codable 行为下会被 encode 时 drop（详见 MAJOR-1），目前没自动化测试能抓到这个 bug。

### 企业管理系统贴谱性

13 实体 DTO 对企业管理系统垂直的覆盖：

- **Initiative.kpis** 数组带 `selected: boolean` ——支持"候选 KPI 池 + 选定子集"的常见 OKR 工作流。fixture initiative.json 用真实业务文案（Q2 客户档案模块改造、合规审核通过率）展示这块用法，**贴谱**。
- **Issue.source** 包含 `inbox`（碎想未分类）+ `ideas_md / git_log / telemetry` 等多入口——对应"碎想入口 + 多渠道反馈"垂直需求，**贴谱**。
- **Artifact.metadata** 在 fixture 用 `businessEntities / permissionMatrix / approvalSteps / reportSchemas` 演示，正好命中"企业管理系统 = 实体 + 表单 + 权限 + 审批 + 报表"五件套。这是契约 #2 比 #1 更靠近垂直的部分。
- **Methodology.appliesTo: claude-web | enterprise-admin | universal** 把方法论按垂直分桶——M0/M1 启动 enterprise-admin dogfood 时，方法论文件可以按 appliesTo 过滤而不混淆。
- **Decision.options 数组 + chosenOption** 支持"通过 / 驳回 / 需更多信息"等 enum-style 决议——fixture decision.json 演示得很清楚。

**贴不上的地方**：

- **Run.cost 是 USD 单位 + REAL 类型**（DDL 同步），但 enterprise-admin 真实业务可能有"按部门 / 项目编码 / 客户" attribution，光一个 cost 字段不够。这个不是 M-1 该管的事，M2 Retrospective.costSummary 才会用上。但 RetrospectiveDtoSchema 的 `costSummary: { totalUsd, byStage, byModel }` 已经把 by_stage / by_model 切片做了，**够用**。
- **Issue.assignee / Issue.dueDate** 不存在。企业内部任务流通常需要"指派给谁、何时到期"。M-1 不强求，但 IDEAS.md 应该记一下，M1+ 加列时是 minor bump。
- **没有 Workflow / Approval-chain 实体**。审批流是企业管理系统核心，但当前用 Decision 单点决议表达；多步审批（一审 → 二审 → 终审）目前需要每步开一个 Decision row + 用户手动串。这个是 methodology 文件该解决的，不是 schema 该解决的，**OK**。

**最脆弱假设**：**Artifact.metadata** 的 typed schema 推到 M2，但 spec 类 Artifact 已经在 fixture 里塞了 4 个企业字段。M2 升 typed 时如果发现 `permissionMatrix` 应该用 `[Role: Set<Action>]` 而不是 `Record<Role, string[]>`，M-1 的"先这么写"的代价就显化了。这个风险已经在源码注释承认，**接受**。

### 风险遗漏

按"4 维 lens 不重叠 reviewer-cross 的安全/正确性"原则，下面只列架构层风险：

1. **iOS 老版本撞上后端新字段的实际行为没在 M-1 字面化**。ADR-0011 写了"minClientVersion 检测 + fallback config"，但具体到协议字段层面，Swift Codable 在缺字段（minor bump）/ 多未知字段（同样 minor bump）时的行为没在 ADR / 协议 doc 里写明。事实上 Swift 默认 JSONDecoder 对**未知字段**是宽容的（自动 ignore），对**缺必需字段**是严格的（throw）；这是协议方向 1.0 → 1.1 minor bump 时"加可选字段不破坏老 iOS"的关键前提，但没在协议 doc 写下来。详见 MAJOR-2。
2. **HarnessEvent 的 Swift discrimination 不抗未知 kind**（MAJOR-3）。
3. **AuditLogEntry.before/after = null 的 Swift round-trip 漂移**（MAJOR-1）——TS 端 `.nullable()` 保留 `null`；Swift 端 Optional `[String:AnyCodable]?` 在 encode 时 drop nil 字段。fixture audit-log-entry.json 显式带 `"before": null`，TS round-trip 通过，但 Swift round-trip（M-1 Phase 7 才做）会失败：原始 JSON 有 `"before": null`，Swift decode 后 before = nil，Swift encode 后 JSON 没有 `before` 这一 key。这破坏了 §6 的"跨端 round-trip 不变量"。
4. **HarnessEvent 的 type 字段是 `discriminatedUnion("kind", ...)` 而 type 是 literal**——Zod discriminated union 在 wire 上不会强校验 `type === "harness_event"`（因为 discriminator 是 `kind` 不是 `type`）。如果客户端漏写 type 或写错 type，Zod 会 parse 通过；只有 Swift 那边手写 init 才会校验 `type === "harness_event"`。两端在"协议入口校验"上的严格度不一致。
5. **协议版本字符串只有 `"1.0"` 一个常量、不是结构化 SemVer**。MIN_CLIENT_VERSION 比较时是字符串字典序还是 SemVer？没有比较函数也没有 ADR 约定。0.9 < 1.0 字典序对，但 1.10 vs 1.9 字典序错。M0 引入网络握手时必须先有 SemVer 比较器。

## 详细发现（除 BLOCKER 外）

### [MAJOR-1] AuditLogEntry.before/after = null 在 Swift round-trip 会丢字段

- **Where**：`packages/ios-native/Sources/ClaudeWeb/HarnessProtocol.swift:296-305` + fixture `audit-log-entry.json:7`
- **Lens**：架构可行性 / 跨端 round-trip 不变量
- **Issue**：TS Zod 用 `z.record(z.unknown()).nullable()`，wire 上保留 `"before": null`。Swift 用 `let before: [String: AnyCodable]?`，默认 JSONEncoder 在 encode optional nil 时**省略 key**而不是写 `null`。fixture 强制带 `"before": null`，TS round-trip 通过（验证过 vitest 42/42 绿），但 Swift round-trip（M-1 Phase 7 才上）必然失败：原始 JSON `{"before": null, ...}` → Swift struct → encode → `{...}`（before 这一 key 没了）。这破坏了 §6 跨端 round-trip 不变量。
- **Suggested fix**：两条路任选其一，**强烈推荐前者**：
  - **(A) 改 wire 约定**：HARNESS_PROTOCOL.md §1 写"可选字段在 wire 上完全省略（不发送 `null`）"。AuditLog 的 before/after 在"无前态"时省略 key，不发 null。fixture 改成 `{ts, actor, op, table, id, after: {...}}`，删除 `"before": null`。Zod 改 `.optional()` 不是 `.nullable()`。Swift 现状不动。这是与 §1"可选字段在 wire 上完全省略" 的全文一致写法。
  - **(B) Swift 自定义 Codable**：给 AuditLogEntry 写自定义 encode(to:) 显式调 `try c.encodeIfPresent + try c.encodeNil(forKey:)`；fixture / TS 不动。代价是每个 nullable 字段都要自定义，未来加字段成本高。

### [MAJOR-2] ADR-0011 没敲"老 iOS + 新字段"的最小协议层行为

- **Where**：`docs/adr/ADR-0011-server-driven-thin-shell.md` Decision 段（line 26-35）
- **Lens**：架构可行性 / 风险遗漏
- **Issue**：ADR-0011 的 Decision 说了 5 条 "M0 时敲" 的项（endpoint、HarnessStore、WS 推送、版本协商、离线兜底），但没敲"协议字段层面 1.0 → 1.1 minor bump 时老客户端做什么"。`HARNESS_PROTOCOL.md §1` 的 "枚举增减按 ADR-0015 管理" 也只说了枚举，没说字段。`LEARNINGS.md #1` 第一条规则 "M0 必须先固定 schema version、minClientVersion、fallback 行为、未知字段处理、旧客户端展示策略"——M-1 没固定的话，M0 还得回头改。
- **Suggested fix**：在 ADR-0011 Decision 段加一句最小行为契约：
  > "minor bump 加可选字段时，老 iOS 默认 JSONDecoder 自动忽略未知字段，UI 不显示新字段；缺新字段不视为 decode 失败。major bump 改语义时，minClientVersion 拒绝服务，回退打包内 fallback。"

  这一句话 30 秒能写完，但锁住了"加字段不破老客户端"的关键前提。

### [MAJOR-3] HarnessEvent Swift 端不抗"未知 kind"，事实上把 minor bump 变成 major

- **Where**：`packages/ios-native/Sources/ClaudeWeb/HarnessProtocol.swift:329-346`（init(from:) switch kind 默认 throw）
- **Lens**：架构可行性 / 风险遗漏
- **Issue**：Swift 的 HarnessEvent.init(from:) 用 switch kind 处理 7 个 kind，default 直接 throw decoding error。M0 / M1 要加 `harness_event { kind: "issue_created", ... }` 等新 kind 时，老 iOS 立即 fail——这违反了 "minor bump 加 enum 值老客户端 graceful skip" 的 ADR-0015 §3.1 约定。protocol 这层把"加 event kind"事实上升级成了 major bump（必须强制重装）。
- **Suggested fix**：default 分支改成"返回一个 `unknown(kind: String, raw: [String:AnyCodable])` case"或者直接不抛 `.dataCorruptedError` 而是 throw 一个 well-known sentinel error，让上层 WS 派发处理函数 catch + telemetry log + 跳过。最小改动是加一个 `case unknown(kind: String)`，default 分支构造 unknown，encode 端跳过（或保留原 JSON）。
- **为什么不算 BLOCKER**：M-1 范围只定义 7 个 kind 的骨架，事实上没有"现在就要 minor bump"的需求；M0 引入第一个新 kind 时再修也来得及。但应该在源码上方加 TODO + 进度文档登记。

### [MAJOR-4] 协议版本字符串没有 SemVer 比较器，握手时谁来比

- **Where**：`packages/shared/src/harness-protocol.ts:28-29`
- **Lens**：架构可行性
- **Issue**：HARNESS_PROTOCOL_VERSION = "1.0"、MIN_CLIENT_VERSION = "1.0"，两边都是裸字符串。HARNESS_PROTOCOL.md §5 说"clientVersion >= MIN_CLIENT_VERSION → 正常"——`>=` 怎么算？字符串字典序还是 SemVer？JS 字符串 `"1.10" >= "1.9"` 是 false（字典序），但 SemVer 上 `1.10 > 1.9`。M0 时如果直接写 `clientVersion >= MIN_CLIENT_VERSION` 就埋雷。
- **Suggested fix**：M-1 加一个 `compareProtocolVersion(a: string, b: string): -1 | 0 | 1` 函数 + 测试，或者明确文档"M-1 内 SemVer 形式锁定为 `^\d+\.\d+$` 不允许 patch / pre-release"，并写出比较规则。后者改动小一点。

### [MINOR-1] HarnessEvent 顶层 type 字段未被 Zod discriminator 校验

- **Where**：`packages/shared/src/harness-protocol.ts:343-351`
- **Lens**：架构可行性
- **Issue**：Zod 用 `discriminatedUnion("kind", ...)`，每个分支里 `type` 是 `z.literal("harness_event")`。Zod parse 通过的前提是 `type === "harness_event"`（每个 branch 都校验）——这条其实是工作的，我多虑了一半。但反过来说，如果未来加 `type: "claude_message"` 等其他顶层消息类型，**两个 discriminatedUnion 怎么共存**？protocol.ts 已有的 SDKMessage 是不是也要并入这套？
- **Suggested fix**：在 §7（与 protocol.ts 关系）加一段："harness_event 的 type 字段是与 protocol.ts 老消息体的 namespace 隔离手段；未来如果合并两个 protocol，应该把所有顶层消息合并成一个 discriminatedUnion('type', ...)，每个 type 内部再 discriminatedUnion('kind', ...)。"

### [MINOR-2] fixture/README.md 没说 audit-log-entry 的 `before: null` 是 wire 约定的关键测试样本

- **Where**：`packages/shared/fixtures/harness/README.md`
- **Lens**：架构可行性
- **Issue**：README 把 audit-log-entry.json 列在"系统 messages"，但没说明 fixture 中的 `before: null` 是为了显式测试 nullable 字段的 round-trip。如果将来有人改 fixture（比如把 before 改成 `{}`）测试照过，但实际上失去了 nullable 测试意义。
- **Suggested fix**：README 加一行："`audit-log-entry.json` 故意保留 `before: null` 以验证 nullable 字段 round-trip 不变量。修改时勿删 null。"（顺便参考 MAJOR-1 决定是否改约定）

### [MINOR-3] Swift `worktreePath` decode 缺失字段时是 nil 而 TS Zod 是 undefined

- **Where**：`HarnessProtocol.swift:195` + `harness-protocol.ts:208`
- **Lens**：架构可行性
- **Issue**：TaskDto.worktreePath 在 fixture task.json 中省略。TS Zod `.optional()` parse 后是 `worktreePath: undefined`；Swift decode 后是 `worktreePath: nil`。两边语义等价、都不在 wire 上发送，OK。但要小心 JSON.stringify 在 TS 端会把 `undefined` 字段省略（一致），而 JSONEncoder 在 Swift 端 nil Optional 也省略（一致）——两边都一致。这条**不是 issue**，写出来作为对自己的反向校验留档。

### [MINOR-4] `tokensIn / tokensOut / cost` 单位 / 精度无契约

- **Where**：`harness-protocol.ts:241-243` (RunDtoSchema)
- **Lens**：企业管理系统贴谱性 / 风险遗漏
- **Issue**：`cost: z.number()` 没说货币、精度（USD？精度到 4 位？）。Run fixture 用 `0.0712`——好像是 USD 4 位小数，但协议没明文。M2 Retrospective 把 `byModel: Record<string, number>` 拼起来时如果不同 row 单位不一致会出错。
- **Suggested fix**：HARNESS_PROTOCOL.md §1 加一行 "金额字段统一 USD，精度由 store 层 round 到 4 位小数"。或在 RunDto 的 zod 注释里加。
