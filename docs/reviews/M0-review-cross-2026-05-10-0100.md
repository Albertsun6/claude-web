## Findings (BLOCKER/MAJOR/MINOR with file:line)

### B1 [BLOCKER] M0 session migration collides with the harness migration stream

**Where**:
- `packages/backend/src/migrations/0004_m0_sessions.sql:1-6`
- `docs/adr/vessel/ADR-006-schema-evolution.md:32-36`
- `packages/backend/src/harness-store.ts:119-122`
- `packages/backend/src/harness-store.ts:128-178`
- `packages/backend/src/harness-store.ts:31`

**Issue**: M0 adds `0004_m0_sessions.sql` to the same `src/migrations/` directory used by `openHarnessDb()`. The harness runner filters all `NNNN_*.sql`, sorts them, and applies pending files. That means the next `openHarnessDb()` will apply the M0 `sessions/intents/skill_invocations` schema to `harness.db`, advance `PRAGMA user_version` to 200, and diverge from `HARNESS_SCHEMA_VERSION = 102`.

**Why this is a blocker**: This is a hidden migration-runner interference, not an incomplete stub. It can corrupt the harness schema timeline and also consumes the exact `0004` filename ADR-006 reserves for future `workflow_state` v103.

**Suggested fix**: Move the memory DB migration out of the harness migration stream, e.g. `src/memory/migrations/0001_m0_sessions.sql`, or change `session-store.ts` to load an embedded/private memory migration path. Keep harness `src/migrations/0004_*` reserved for `0004_workflow_state.sql` v103.

### B2 [BLOCKER] TraceWriter does not perform mandatory redaction before persistence

**Where**:
- `packages/backend/src/observability/trace-writer.ts:7-8`
- `packages/backend/src/observability/trace-writer.ts:41-49`
- `packages/backend/src/observability/trace.ts:88-98`
- `docs/design/trace-redaction-spec.md:51-85`
- `docs/design/trace-redaction-spec.md:142-153`
- `docs/design/FRAMEWORK.md:536-544`
- `docs/design/FRAMEWORK.md:921-923`

**Issue**: `FileTraceWriter.write()` only runs `TraceEventSchema.parse(event)` and writes `JSON.stringify(event)` directly. The implementation comment explicitly says redaction is not done yet, while the interface and FRAMEWORK require `write()` itself to redact payloads according to the spec.

**Why this is a blocker**: The spec marks trace redaction as mandatory for M0 acceptance and hard-trigger secrets/privacy protection. The writer can persist `payload.user_prompt`, token-like strings, private paths, headers, cookies, and API response bodies in trace JSON.

**Suggested fix**: Add redaction inside `TraceWriter.write()` before schema persistence: field-path blacklist, content-pattern replacement with `***-redacted-<hash6>***`, path allowlist handling, and large-output summary/artifact handling as specified. Do not rely on callers.

### m1 [MINOR] Trace directory root permissions are not guaranteed when parent directories already exist

**Where**:
- `packages/backend/src/observability/trace-writer.ts:45-49`
- `docs/design/trace-redaction-spec.md:129-138`

**Issue**: `mkdirSync(dir, { recursive: true, mode: 0o700 })` sets mode for newly created directories, but it does not repair permissions if `<DATA_DIR>/traces` or an existing `<trace_id>` directory was created earlier with broader permissions.

**Why this matters**: The spec requires trace directories to be owner-only. This is lower priority than B2 because current fresh-create behavior is correct for the leaf directory, but reused directories can silently violate the permission invariant.

**Suggested fix**: After ensuring the trace directory exists, call `chmodSync(dir, 0o700)` for the trace-id directory. Consider also creating/chmodding the `traces` root.

### m2 [MINOR] `ToolRegistry` implementation interface is narrower than FRAMEWORK

**Where**:
- `packages/backend/src/interfaces/skill.ts:42-45`
- `docs/design/FRAMEWORK.md:123-128`
- `packages/backend/src/orchestrator.ts:75-80`

**Issue**: FRAMEWORK defines `ToolRegistry` with both `get()` and `list()`, but the implementation only includes `get()`. The M0 orchestrator stub passes only `{ get: () => null }`.

**Why this matters**: This is not a blocker for EchoSkill, but it is a type-contract drift that can hide until the first real Skill expects `tools.list()`.

**Suggested fix**: Either add `list(): Tool[]` to `interfaces/skill.ts` and pass `list: () => []` in M0, or update FRAMEWORK if `list()` was intentionally dropped.
