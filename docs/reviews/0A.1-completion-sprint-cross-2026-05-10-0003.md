## Findings

### B1 [BLOCKER] Trace ID / span ID / 子进程传播协议同时存在三套互斥定义

**Where**: `docs/design/FRAMEWORK.md` §5 `TraceEventSchema`, §5.5 OTEL 映射, §5.6 W3C Trace Context；`docs/roadmap/ROADMAP.md` 0B-8 / NFR-O1  
**Lens**: 1, 2  
**Issue**: §5 代码仍定义 `trace_id: z.string().uuid()`、`span_id: z.string().uuid()` 和 `ENV_KEYS = VESSEL_TRACE_ID...`；但 §5.5 又说 M0 起改成 OTEL 32-hex / 16-hex；§5.6 又说 M0 改用 `TRACEPARENT` 替代自定义 env vars。  
**Why this is a blocker**: 0B-8 马上要实现 `observability/trace.ts`。如果按 §5 实现就是 UUID + 自定义 env；按 §5.5/§5.6 实现就是 OTEL hex + W3C traceparent。trace replay、子进程传播、Swift/TS fixture 会从第一版开始分裂，后面迁移成本高。  
**Suggested fix**: 在 FRAMEWORK 中只保留一个 M0 契约。建议直接把代码块改成 M0 最终形态：`trace_id` 32 hex、`span_id` 16 hex、`parent_span_id` 16 hex nullable，删除 `ENV_KEYS` 或改成 `TRACEPARENT` + `VESSEL_CONVERSATION_ID` + `VESSEL_RUN_ID`。ROADMAP 0B-8 acceptance 同步写成 W3C traceparent。

### M1 [MAJOR] ML worker 协议在 FRAMEWORK 与 ADR-012 内部互相打架

**Where**: `docs/design/FRAMEWORK.md` §4.2；`docs/adr/vessel/ADR-012-language-typescript-with-ml-worker.md` §5 / §6  
**Lens**: 1, 3  
**Issue**: FRAMEWORK 已锁定 HTTP loopback + OpenAI-compatible API，`EmbeddingClient.embed(input: string | string[], opts?)`；ADR-012 §5 仍写 `JSON-RPC 协议`，§6 又写旧接口 `embed(texts: string[])`、`transcribe(wav: Buffer)`、`synthesize(text, { rate })`。  
**Why this matters**: 0B-9 要落 `ml-worker/types.ts` stub，实施者不知道按哪个签名写。更严重的是 HTTP worker 和 JSON-RPC worker 的 manager、health、错误归因、测试 fixture 都不同。  
**Suggested fix**: ADR-012 §5 删除 “JSON-RPC 协议”，改成 “HTTP loopback client”。ADR-012 §6 直接引用 FRAMEWORK §4.2 的最终 TS interface，字段名保持一致：`input`、`opts.model`、`speed`、streaming API 是否进 M0 也要明确。

### M2 [MAJOR] Soul Spec 已改 4 sibling，但 ROADMAP 仍保留旧单文件和旧字段 acceptance

**Where**: `docs/design/FRAMEWORK.md` §6.1–§6.5；`docs/roadmap/ROADMAP.md` M2-Soul-4 / M2-Soul-6  
**Lens**: 2, 3  
**Issue**: FRAMEWORK 规定 `templates/soul/<template>/{SOUL,STYLE,SKILL,MEMORY}.md`，schema 字段是 `identity` / `values` / `guardrails`；ROADMAP 仍写 `templates/soul/{jarvis-style,friday-style,blank}.soul.md`，并要求 `vessel-core soul show-prompt` 输出含 `personality` 字段。  
**Why this matters**: 这是跨文档 contract drift。M2-Soul 实施会出现 parser、template、acceptance 三者互相不匹配。  
**Suggested fix**: ROADMAP M2-Soul-4 改为三个 template 目录，每个目录含 4 sibling 文件。M2-Soul-6 acceptance 改成输出含 `identity.name`、`values`、`guardrails`，不要再引用 `personality`。

### M3 [MAJOR] AgentResult / ArtifactSchema 用 optional 字段表达状态，无法强制关键不变量

**Where**: `docs/design/FRAMEWORK.md` §2.1 `AgentResult`；§8 `ArtifactSchema`  
**Lens**: 1  
**Issue**: `AgentResult.status='success'` 时 `artifact` 仍是 optional；`paused` 时 `resumeToken` 仍是 optional；`failed` 时 `error` 仍是 optional。`ArtifactSchema` 也允许 `type='file'` 但没有 `filePath`，或 `type='text'` 但没有 `content`。  
**Why this matters**: 文档注释说“成功路径必须是 `{ status: 'success', artifact }`”，但接口本身不 enforce。后续 TS、SQLite、Swift Codable fixture 都会接受非法状态，错误会推迟到运行时。  
**Suggested fix**: 改成 discriminated union / Zod discriminatedUnion。比如 `AgentResult = SuccessResult | PausedResult | CancelledResult | FailedResult`；`ArtifactSchema` 按 `type` 分支要求 `content` 或 `filePath`。

### M4 [MAJOR] 权限 pathAllowlist 没有 canonicalization / symlink escape 规则

**Where**: `docs/design/FRAMEWORK.md` §2.3 `PermissionScope.pathAllowlist`；§7 manifest 示例；§10 NFR-P1 / NFR-P3  
**Lens**: 4  
**Issue**: 文档只说 `glob 或绝对前缀`，manifest 示例用 `~/Desktop/Vessel/instance/workspace/`。没有定义 `~` 展开、相对路径、大小写文件系统、symlink、`../`、realpath 后缀检查。  
**Why this matters**: 路径白名单是 Vessel 权限核心。如果只做字符串前缀匹配，workspace 内 symlink 可以指向 `~/.ssh` 或其他敏感目录，NFR-P1/P3 会被绕过。  
**Suggested fix**: 在 FRAMEWORK §2.3 或 M1B acceptance 加硬规则：所有路径先 `realpath` canonicalize；allowlist 也 canonicalize；禁止跟随逃出 instance root 的 symlink；`~` 只允许配置加载阶段展开；测试覆盖 `../` 和 symlink escape。

### M5 [MAJOR] OTEL SDK 引入与 ROADMAP “不引 OpenTelemetry 库”互相矛盾，且有 recency-bias 风险

**Where**: `docs/design/FRAMEWORK.md` §5.7；`docs/roadmap/ROADMAP.md` §11.2 “OpenTelemetry-lite 12 字段”；`docs/research/0A-completion-sprint-prior-art-2026-05-09.md` §10 不确定点 2  
**Lens**: 5  
**Issue**: FRAMEWORK §5.7 推荐 M0 用 `@opentelemetry/api` + `@opentelemetry/sdk-node`，但 ROADMAP §11.2 仍写 “observability/trace.ts 自实现（不引 OpenTelemetry 库）”。这像是 A7 调研后局部 pivot，没有完成全局一致化。  
**Why this matters**: 这既是实施冲突，也是集体盲区：因为 OTEL GenAI 是近期标准，文档可能从“轻度映射兼容”滑向“M0 直接引 SDK”，但没有重新评估 M0 复杂度、redaction hook、trace 文件权限和 Swift fixture 成本。  
**Suggested fix**: 二选一写死：  
- M0 自实现 12 字段 + OTEL 命名 alias，不引 SDK；v1+ exporter 再引。  
- 或 M0 直接引 OTEL SDK，则 ROADMAP §11.2、0B-8、trace-redaction-spec acceptance 全部同步，并加一个最小 redaction hook 设计。

### m1 [MINOR] Intent attachment base64 没有大小上限和 artifact_refs 降级路径

**Where**: `docs/design/FRAMEWORK.md` §8 `IntentSchema.attachments[].dataBase64`  
**Lens**: 1, 4  
**Issue**: Trace payload 有 4KB 限制，但 Intent attachments 的 base64 没有限制，也没有说明大文件应先落盘再传 `artifact_refs`。  
**Suggested fix**: 给 attachment 加 `sizeBytes`、max size，或规定超过阈值必须走 workspace file reference。iOS / HTTP upload acceptance 也应覆盖超大附件拒绝路径。

### m2 [MINOR] ADR-012 说 `sqlite-vec` 属于 ML 任务，但又说它不是子进程，分类容易误导

**Where**: `docs/adr/vessel/ADR-012-language-typescript-with-ml-worker.md` Decision §2 表格  
**Lens**: 3  
**Issue**: 表格标题是 “ML 任务走子进程 worker 边界”，但 `向量库 sqlite-vec` 行写 “不是子进程”。这不是功能错误，但会让边界判断变模糊。  
**Suggested fix**: 把表格拆成 “ML inference worker” 和 “local vector storage”。明确 sqlite-vec 是 SQLite extension，不归 ML worker lifecycle 管。
