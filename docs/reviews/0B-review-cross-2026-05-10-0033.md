## Findings

### BLOCKER-1: stale `CLAUDE_WEB_*` env guard is defined but never invoked
- File: `packages/backend/src/index.ts:1`
- Issue: `startup-env-check.ts` defines `checkRenamedEnvVars()` and says stale `CLAUDE_WEB_*` env vars must alert + exit, but the backend entrypoint import block does not import or call it before booting the server. `auth.ts` and `data-dir.ts` now read only `VESSEL_*`, so a launchd/profile still using old `CLAUDE_WEB_TOKEN` or `CLAUDE_WEB_ALLOWED_ROOTS` will silently start with no auth token / no allowlist.
- Why blocker: This is a security regression in the 0B rename path. The new guard is the only mitigation for “no code-side fallback”; because it is not wired, the app can be exposed unauthenticated after rename while the user believes the old token is still active.
- Suggested fix: Import and call `checkRenamedEnvVars()` at the very start of backend startup, before registering routes or serving. Add a small test that spawns the backend with `CLAUDE_WEB_TOKEN=x` and asserts non-zero exit plus the migration message.

### MAJOR-1: 5 interface stubs do not match FRAMEWORK signatures byte-for-byte
- File: `packages/backend/src/interfaces/agent.ts:18`
- Issue: FRAMEWORK §2.1 requires `import type { Intent, Artifact, SessionId } from '@vessel/shared';`, but the stub comments that import out and replaces all three with local placeholders (`unknown` / `string`). `skill.ts` repeats the same pattern for `Intent` and `Artifact`.
- Why major: The 0B deliverable is the interface contract. Using `unknown` lets code compile while erasing the contract that downstream Agent/Skill implementations are supposed to satisfy, and it hides the fact that `@vessel/shared` does not yet export the promised `Intent` / `Artifact` / `SessionId` types.
- Suggested fix: Either add the FRAMEWORK §8 `IntentSchema` / `ArtifactSchema` / `SessionId` exports to `@vessel/shared` now and import them from the stubs, or explicitly amend FRAMEWORK to say 0B uses temporary placeholders. Do not leave comments claiming the FRAMEWORK import while exporting weaker types.

### MAJOR-2: `TraceContext` implementation drifts from FRAMEWORK §5
- File: `packages/backend/src/observability/trace.ts:65`
- Issue: FRAMEWORK §5 defines `TraceContext` with `session_id`, but the implementation exports `conversation_id` plus an extra required `trace_flags`. The TraceEvent schema still uses `session_id`, and all interface contexts (`Agent.traceCtx`, `SkillContext.traceCtx`, `ToolContext.traceCtx`, `CodingDriver` ctx) now depend on the implementation shape.
- Why major: This is a low-level type contract split: code using `traceCtx.session_id` per FRAMEWORK will fail typecheck, while code using `conversation_id` will drift away from the TraceEvent schema and OTEL mapping table. It is exactly the kind of hidden narrowing/field-name mismatch that later creates adapter glue everywhere.
- Suggested fix: Make `TraceContext` match FRAMEWORK exactly (`session_id`) or update FRAMEWORK §5 everywhere to use `conversation_id`. If W3C trace flags are required, add them to FRAMEWORK as an explicit field rather than silently adding them only in code.

### MAJOR-3: migration script copies live SQLite database files without a consistency guard
- File: `scripts/migrate-eva-to-vessel.ts:36`
- Issue: The script copies `harness.db`, `harness.db-wal`, and `harness.db-shm` as ordinary files. There is no check that the Eva backend is stopped, no SQLite backup API, no WAL checkpoint, and no temp-dir + atomic rename around the destination set.
- Why major: The script is non-destructive to the source, which is good, but it can still produce an inconsistent Vessel database if Eva is running or if the WAL changes between file copies. That makes the migration “safe from deletion” but not “safe from corrupt snapshot.”
- Suggested fix: Before copying SQLite files, require Eva/vessel backend not running or run a SQLite backup/checkpoint path. Copy into a temporary destination directory, verify the DB opens, then atomically rename into place. At minimum, print a hard warning and fail unless `--allow-live-copy` is passed.

### MINOR-1: ML stream method optionality does not match FRAMEWORK §4.2
- File: `packages/backend/src/ml-worker/types.ts:28`
- Issue: FRAMEWORK §4.2 shows `transcribeStream(...)` and `synthesizeStream(...)` as required interface methods. The stub makes both optional with `?`.
- Suggested fix: Decide one contract. If streaming is post-M2 optional, update FRAMEWORK to mark it optional. If it is part of the client contract, remove `?` from the stub.

### MINOR-2: 0B tsc acceptance cannot be verified from current scripts
- File: `package.json:6`
- Issue: 0B acceptance says `pnpm tsc --noEmit` exits 0, but root `package.json` has no `tsc` or `typecheck` script. Per-package `tsc` can be run manually, but there is no single documented root command matching the acceptance sentence.
- Suggested fix: Add a root script such as `"typecheck": "pnpm --filter @vessel/shared exec tsc --noEmit && pnpm --filter @vessel/backend exec tsc --noEmit && pnpm --filter @vessel/frontend exec tsc -b"` and update acceptance to that exact command.
