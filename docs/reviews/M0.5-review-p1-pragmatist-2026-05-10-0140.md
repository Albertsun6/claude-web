### F-P1 [MAJOR] `SkillContext.abortSignal` 已用 ✅，但 `tools/memory` 假手对象 M0.5 仍未拆，M0 pragmatist 的建议被忽略

**严重程度**：MAJOR（架构债务被沿用；M0 review 已点出，M0.5 仍未修）

**用户 review focus #1 直接问**：`SkillContext.abortSignal 是 M0.5 真需要还是为 v1 准备?`

**事实**：
- `interfaces/skill.ts` L40-42:
  ```typescript
  abortSignal: AbortSignal;  // 注释：Skills MUST forward to any subprocess they spawn
  ```
- `orchestrator.ts` L94: `const abortSignal = input.abortSignal ?? new AbortController().signal;`
- `skills/coding.ts` L50: 真正传给 driver
- `drivers/cli-runner-driver.ts` L60/89/90/123: 真正消费——`addEventListener('abort', ...)` + `signal.aborted` short-circuit

**回答**：`abortSignal` **是 M0.5 真需要**，**不是 v1 准备的**。
- M0.5 ROADMAP 明示需要 process-group cancel 路径（user focus #3）
- `test-coding-driver.ts` Test 4 直接验证 `setTimeout(() => abortCtl.abort(), 50)` → status=cancelled
- 如果不放进 SkillContext，CodingSkill 必须自己开个 side-channel 拿 abort signal——更难看
- 评级：**接口这一加是合理的、不是 over-engineer**

**但 M0.5 的真正问题是**：`tools / memory / workspaceDir` 三个假手对象**仍然在 orchestrator.ts L102-104 留着**：

```typescript
tools: { get: () => null },          // M0.5 无 Tool（M1B+ 接 MCP）
memory: {} as never,                 // M0.5 不接 Memory full surface（M1C+）
workspaceDir,
```

M0 pragmatist (F-P3) 已经点过这条 + 给出修复方案（接口 fields → optional），M0.5 PR 没采纳。**这一次必须明确**：

1. **M0.5 的 CodingSkill 不读 ctx.tools / ctx.memory**——echo 也不读——所以**当前两个 skill 都靠这两个假手对象通过 TS 检查**。`memory: {} as never` 是 type-system escape hatch，M1C 加 Memory.read() 时**任何 M0.5 写的 Skill 会运行时炸**（当然 M0.5 只有 echo + coding，但 SkillContext 的语义已被毒化）。

**给 owner**：M0 pragmatist F-P3 的判定 unchanged；M0.5 没修是欠债，**arbiter 应该把这条升级到 M1B 前必修**。

### F-P2 [MAJOR] `cli-runner-driver.ts` workspace 双重 realpath 是过度防御

**严重程度**：MAJOR（user focus #2 直接质问；引入新故障面）

**用户 review focus #2 直接问**：`workspace iso check 用 realpath 是否过度防御？`

**事实**（cli-runner-driver.ts L62-81）：

```typescript
const expected = join(workspaceRoot, spec.runId);
if (!existsSync(expected)) mkdirSync(expected, { recursive: true, mode: 0o700 });

const wsReal = realpathSync(expected);              // 1st realpath
const rootReal = realpathSync(workspaceRoot);
if (!wsReal.startsWith(rootReal + sep) && wsReal !== rootReal) {
  throw new Error(`workspace escape: ${wsReal} not under ${rootReal}`);
}
// ...comment says "Reject if caller passed a workspace path whose realpath disagrees with..."
if (realpathSync(spec.workspace) !== wsReal) {       // 2nd realpath, on caller's input
  throw new Error(
    `spec.workspace realpath must equal ${wsReal} (got ${spec.workspace}); ...`,
  );
}
```

**逐步质问**：

1. **第一处 `realpathSync(expected)` + `startsWith(rootReal + sep)`**：完全合理。这条防"`workspaceRoot/<runId>` 被某个攻击者通过 symlink 指向 `/etc/passwd`"——但**个人单机场景**下，谁在 `<DATA_DIR>/workspace/` 下放 symlink？只能是用户自己。即便如此，多一层 belt-and-braces 成本 = 1 syscall + 5 行代码 ✅。**这层留**。

2. **第二处 `realpathSync(spec.workspace) !== wsReal`**：**完全没必要**。
   - **谁是 caller**？只有 CodingSkill (`skills/coding.ts`)。
   - **caller 怎么拿 spec.workspace**？`ctx.workspaceDir`，由 orchestrator.ts L93 通过 `workspaceFor(runId)` 计算——**就是 `cli-runner-driver.ts` 自己 export 的同一个 helper**！
   - 所以 spec.workspace == workspaceFor(runId) == join(resolveWorkspaceRoot(), runId) == 与 driver 内部 `expected` **同一个字符串**——除非 caller bypass orchestrator 直接构造 spec。
   - 这条 check 实质上是"**driver 怀疑 caller 撒谎**"。**caller 是 skill，skill 是 driver 注入的，driver 是 orchestrator 实例化的**——同一个进程内的同一个团队代码。
   - **个人单机硬约束**：没有"恶意 third-party Skill"威胁面。
   - **故障面引入**：如果 `spec.workspace` 不存在（caller 还没 `mkdir`），`realpathSync(spec.workspace)` 抛 ENOENT，先于 driver 内部的 `mkdirSync(expected, ...)`——错误信息不诚实（caller 看到的是 ENOENT，不是"contract violation"）。

**修复**：删除 L77-81 共 5 行。第一层 check（L71-73）已经是充分的 isolation。

**给 owner**：L77-81 是用户 focus #2 的真正答案。**workspace iso 没有 over-engineered**（第一层是必需的）；**第二层是 over-engineered**（5 行删掉）。

### F-P3 [MAJOR] `redactor SUFFIXES` 14 条暴增 + 包含毁掉 trace 可读性的字段

**严重程度**：MAJOR（user focus #4 直接问；正确权衡 false-positive vs false-negative）

**用户 review focus #4 直接问**：`redactor SUFFIXES 列表是否暴增？正确权衡 false-positive (有用字段被 mask) vs false-negative (敏感字段漏 mask)？`

**事实**：
- M0 pragmatist 看的版本里 `REDACTED_FIELD_SUFFIXES` 是 0 条（M0 trace 只 redact `.user_prompt` / `.cli_args.path` 等显式 path）
- M0.5 添加：

```typescript
const REDACTED_FIELD_SUFFIXES = [
  '.thinking', '.text', '.content', '.input',
  '.command', '.file_path', '.filePath',
  '.structuredPatch', '.originalFile',
  '.description', '.result', '.summary',
  '.message', '.prompt', '.user_prompt',
  '.error_message',
];
```

**逐字段权衡**：

| Suffix | 出现频率（CC stream-json） | 真实包含敏感数据？ | False-positive 损失 | 评分 |
|---|---|---|---|---|
| `.thinking` | 高（assistant blocks） | 是（model 内部推理可能引用 prompt） | 可读性损失 — debug 时全是 `***` | ⚠️ 应保留 |
| `.text` | **极高**（**assistant message text / tool result text / system message text 全用这个 key**） | 部分（assistant text 含模型生成内容；tool result text 含文件内容） | **巨大** — trace 里 99% 用户能看的内容字段都叫 `.text` | ❌ **过度** |
| `.content` | 高（消息体） | 部分 | 大 — 整个 message body 全 mask | ❌ **过度** |
| `.input` | 中（tool_use.input） | 是（含 file_path / command / content） | 中 | ✅ 合理 |
| `.command` | 中（Bash tool） | 是（shell 命令） | 中 | ✅ 合理 |
| `.file_path` / `.filePath` | 中 | 部分（白名单内的应放过） | 但 `.file_path = "/Users/yongqian/Desktop/Vessel/foo.ts"` 会被整段 `***`，**absolute path 白名单失效**——因为 path-suffix 直接 `*** override` 比 string-pattern 早 | ⚠️ 损失白名单 |
| `.structuredPatch` | 中（Edit tool） | 是 | 中 — patch 体含代码内容 | ✅ 合理 |
| `.originalFile` | 低 | 是 | 低 | ✅ 合理 |
| `.description` | 高（tool description / Skill.description） | 否（**纯静态文档字符串**） | **巨大** — Skill.description = 'Spawn coding CLI...' 也被 `***` | ❌ **过度** |
| `.result` | 高（CC final result） | 部分 | 高 — 但**用户最想在 trace 里看的就是 result**，全 mask 等于 trace 没用了 | ⚠️ 应保留 |
| `.summary` | 中 | 部分 | 中 — `payloadSummary` 自己生成的字段也叫 summary，会被自己 mask（递归触发？）| ⚠️ self-fire 风险 |
| `.message` | **极高**（error.message / system.message / 任何字段叫 message） | 部分 | **巨大** — `error.message` 是 trace 调试核心，全 mask 等于 unhelpful | ❌ **过度** |
| `.prompt` | 低 | 是 | 低 | ✅ 合理 |
| `.user_prompt` | 低（M0.5 不直接用） | 是 | 低 | ✅ 合理（已在 PATHS） |
| `.error_message` | 低 | 部分 | 中 | ⚠️ 同 `.message` |

**总评**：`.text / .content / .description / .message / .result` **5 个字段**是 trace 最常见的"信息载荷"——把这些**默认全 mask**，换来的是：
- **False-negative 收益**：CC stream-json 里如果 user prompt 真的混进了 `.text` 字段，会被遮蔽 ✅
- **False-positive 代价**：trace 文件里 80% 内容变成 `***`，`grep -i` 排查 bug 全失效；用户分享 trace 给开发者也看不到任何上下文

**这正是 trace-redaction-spec §3a 没列这些字段的原因**——spec §3a 只列 `payload.user_prompt / payload.cli_args.path / payload.api_response.body` 等**精确路径**。M0.5 的扩展是合理的（CC stream-json 的字段名跟 spec §3a 路径不一一对应），**但 14 条扩出来时没区分 high-risk vs low-risk**。

**修复**（按 false-positive 风险排序删）：

1. **删 `.description`**（纯静态字符串，不可能含 secrets）
2. **删 `.text` 或改成更窄的路径**（如 `.message.content[].text` 这种 OTEL-style path，需要扩 path matcher）；当前粗放匹配会把 `Skill.description` 里嵌套的 text 也吞掉
3. **删 `.message`** 或改成 `.error.message`（更精确）
4. **`.result` / `.summary` / `.error_message`** 保留但加上"**redact 后保留首 100 字符**"的轻量替代——完全 `***` 让 trace 失效
5. **`.file_path` / `.filePath`**：与 PATH_WHITELIST 联动——白名单内 path 应放过，否则白名单形同虚设；当前实现（L119）`isPathRedacted(path) && (...)` 直接返回 `'***'`，没走 `redactString` 的白名单逻辑

**给 owner**：F-P3 是 M0.5 最值得现在修的一条。SUFFIXES 14 → 8（删掉 5 条）+ `.file_path` 走白名单逻辑。

### F-P5 [MINOR] cli-runner-driver.ts 的 stdout 处理：line-by-line redact 在大 stdout 下放大

**严重程度**：MINOR（M0.5 不阻塞；M2+ 量产前必看）

**事实**（cli-runner-driver.ts L131-140）：

```typescript
if (stdoutLines.length > 0) {
  const redacted = stdoutLines.map((line) => {
    try {
      return JSON.stringify(redactPayload(JSON.parse(line)));
    } catch {
      return JSON.stringify({ unparseable_line_redacted: true });
    }
  });
  writeFileSync(stdoutPath, redacted.join('\n') + '\n', { mode: 0o600 });
}
```

**问题**：
1. **内存放大 3-4×**：`stdoutLines: string[]` 累积（原始 string） + `redacted: string[]`（redacted JSON）+ `redacted.join('\n')`（最终 buffer）。CC stream-json 一次跑可达数百行 × 数 KB = 几 MB。M0.5 短任务 OK，M2+ 真任务（refactor 大代码库）会撑大 RSS。
2. **每行 `JSON.parse` + `JSON.stringify` 都要走完整 redactor**：对深度递归对象（`assistant.content[].text` 嵌套很深），CPU 不便宜。
3. **`.write stream` 的写法可以替代**：`fs.createWriteStream(stdoutPath, { mode: 0o600 })` + `.write(redacted_line + '\n')` per-line，让 stdout 行流式刷盘，不在内存累积。

**M0.5 不修无碍**：echo 不走这条；fake driver 也不走（`message_count` 来自 fixture）。真实 CodingDriver 跑 fibonacci 这种小任务，stdout 本来就小（< 几 KB）。

**M2+ 必修**：到 production 跑大任务时，这是 OOM 雷。

**给 owner**：F-P5 列入 M2 backlog，**不在 M0.5 fix 列表里**。
