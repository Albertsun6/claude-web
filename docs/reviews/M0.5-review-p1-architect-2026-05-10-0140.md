### Focus 2 (workspace iso robustness, symlink escape): **PASS-WITH-MAJOR**

`cli-runner-driver.ts:62-81` does the right things:

```typescript
const wsReal = realpathSync(expected);
const rootReal = realpathSync(workspaceRoot);
if (!wsReal.startsWith(rootReal + sep) && wsReal !== rootReal) {
  throw new Error(`workspace escape: ${wsReal} not under ${rootReal}`);
}
if (realpathSync(spec.workspace) !== wsReal) {
  throw new Error(`spec.workspace realpath must equal ${wsReal} ...`);
}
```

Realpath-vs-realpath comparison correctly handles macOS `/var ↔ /private/var`
symlinks. The `wsReal !== rootReal` short-circuit prevents the edge case
where someone passes the workspaceRoot itself as the workspace (since
`startsWith(rootReal + sep)` would fail when wsReal equals rootReal exactly).

**MAJOR-1: TOCTOU race on workspace creation.**
Sequence:
1. Line 67: `mkdirSync(expected, { recursive: true, mode: 0o700 })`. If `expected`
   is a pre-existing symlink to `~/.ssh/`, mkdir-recursive does NOT replace it
   (Node's `mkdirSync` recursive treats existing symlinks-to-directories as
   "exists, no-op"; it does NOT verify the target is under workspaceRoot).
2. Line 69: `realpathSync(expected)` resolves the symlink → returns `~/.ssh/`.
3. Line 71: `wsReal.startsWith(rootReal + sep)` — `~/.ssh/` does NOT start with
   `~/.vessel/workspace/`. **The check rejects.**

So in steady state the check works. **But**: between line 67 (`mkdirSync`) and
line 69 (`realpathSync`), an attacker with local FS access could swap the
just-created directory with a symlink to `~/.ssh/`. This is a low-impact
race (Vessel is single-user; attacker would need shell access). **Mitigation
recommended**: open the directory with `O_NOFOLLOW` (Node 16+ via
`fs.openSync(p, fs.constants.O_NOFOLLOW | fs.constants.O_DIRECTORY)`) BEFORE
realpath, or use `fs.lstatSync(expected)` to assert `!stat.isSymbolicLink()`
before the realpath dance. This is a hardening fix, not a closeout BLOCKER.

**MAJOR-2: workspaceRoot itself is not lstat-checked.**
Line 64-65: `if (!existsSync(workspaceRoot)) mkdirSync(...)`. If
`~/.vessel/workspace` is a pre-existing symlink to `/tmp/evil/`, `existsSync`
returns `true`, mkdir is skipped, `realpathSync(workspaceRoot)` returns
`/tmp/evil/`, and ALL workspaces are silently under `/tmp/evil/`. The
startsWith check still passes locally (wsReal starts with that rootReal), so
the iso check is satisfied — but the workspace itself is outside `~/.vessel/`.
This is again a hardening issue, not strictly a closeout blocker.

### Focus 3 (process group cancel: SIGTERM → 5s → SIGKILL): **PASS-WITH-MAJOR**

cli-runner.ts:91-115 is structurally correct:
- detached spawn → `child.pid` is its own pgid
- `process.kill(-child.pid, 'SIGTERM')` reaches the whole subtree
- `setTimeout(... KILL_GRACE_MS=5000)` escalates to SIGKILL
- `clearTimeout(killTimer)` on `close` and `error` events

`child.pid === undefined` guard (line 96, 107) is correct — would be undefined
if spawn() failed to fork.

**MAJOR-3: Race between vessel-core's hard-exit fallback and cli-runner's
SIGKILL escalation.**

`cli/vessel-core.ts:90-94`:
```typescript
setTimeout(() => {
  try { closeMemoryDb(); } catch { /* ignore */ }
  process.exit(sig === 'SIGINT' ? 130 : 143);
}, 5000).unref();
```

`cli-runner.ts:104-114`:
```typescript
killTimer = setTimeout(() => {
  if (child.exitCode === null && !child.killed) { ... SIGKILL ... }
}, KILL_GRACE_MS); // 5000ms
```

Both timers start at the moment of SIGINT/SIGTERM receipt (vessel-core
schedules its 5s fallback first; the abort fires; cli-runner schedules its
own 5s SIGKILL escalation a microtask later). At T+5000ms BOTH fire. The
race outcomes:
- vessel-core's setTimeout wins → `process.exit(130)` → cli-runner's child
  is orphaned (process group still owned by vessel-core; gets reparented to
  init/launchd; SIGKILL never lands because escalation timer was inside the
  exiting process).
- cli-runner's setTimeout wins → SIGKILL sent → child exits → `runSession`
  promise resolves → `runIntent` returns → main() returns 130 → process.exit(130).

The problem: the **5s budget is the same for both**, but vessel-core's path
needs the cli-runner's path to complete THEN do `closeMemoryDb()`. Recommended
fix: bump vessel-core hard-exit fallback to **6000ms** (or `KILL_GRACE_MS +
1000`) to give cli-runner room to escalate, drain `child.on('close')`, return
the runSession promise, run finally{} cleanup, and let the shutdown handler
in vessel-core call `closeMemoryDb()` properly. Acceptance C-3 of plan
("ctrl-c after 5 seconds pgrep -g returns empty") refers to the CHILD pgroup
clearing, not the parent — but if vessel-core orphans before SIGKILL fires,
the test will FAIL on a slow CI runner.

Verified the acceptance command works on a fast laptop (T+~5050ms), but the
ordering is brittle. NFR-F1 ("5 second budget") is currently best read as
"we hard-exit AT 5s" rather than "everything cleaned up BY 5s" — which is
inconsistent with what M0 acceptance C-3 actually checks.

### Focus 4 (trace 不泄露 prompt / stdout): **FAIL** — BLOCKER + MINOR

#### BLOCKER-1: stdout file is redacted via `redactPayload(JSON.parse(line))` with seed path `'payload'` — the redactor's CC-stream-json suffix list uses `path.endsWith(suffix)`, but `system_init` events have NO matching suffix.

`cli-runner-driver.ts:131-140`:
```typescript
const redacted = stdoutLines.map((line) => {
  try { return JSON.stringify(redactPayload(JSON.parse(line))); }
  catch { return JSON.stringify({ unparseable_line_redacted: true }); }
});
```

`trace-redactor.ts:138-141` `redactPayload` calls `redactValue(value, 'payload')`.
So a CC `system_init` event at the top level looks like:
```json
{ "type": "system", "subtype": "init", "cwd": "/Users/yongqian/Desktop/Vessel/instance/workspace/<runId>", "tools": [...], "session_id": "...", "model": "..." }
```

After `redactValue` walks it:
- `payload.cwd` → string → `redactString` runs path-regex → matches `/Users/yongqian/...`, but `/Users/yongqian/Desktop/Vessel/` IS in PATH_WHITELIST → NOT redacted. **OK** (intended).
- `payload.tools` → array of tool descriptors `[{ name: 'Bash', description: 'Execute...' }, ...]` → `description` field has `.description` suffix → masked. **OK**.
- BUT: tool names like `Bash`, `Read`, etc. do not match any suffix and pass through. Acceptable.

For a CC `tool_use` event:
```json
{ "type": "assistant", "message": { "content": [{"type": "tool_use", "name": "Bash", "input": {"command": "rm -rf ~/private"}}] } }
```
- Path `payload.message.content[0].input` ends with `.input` → masked to `***`. Object form replaced; NESTED `command` value never reached. **GOOD**.

For a CC `text` event (assistant final answer):
```json
{ "type": "assistant", "message": { "content": [{"type": "text", "text": "Here is the code: ..."}] } }
```
- Path `payload.message.content[0].text` ends `.text` → masked. **GOOD**.

For the actual user prompt echoed back:
```json
{ "type": "user", "message": { "role": "user", "content": "<user prompt full text>" } }
```
- Path `payload.message.content` ends `.content` → masked. **GOOD**.

So **the prompt IS redacted in stdout**. The BLOCKER is more subtle:

**The actual leak**: CC stream-json embeds the full `cwd` as a top-level field
on EVERY `system_init` and inside `tool_use.input.cwd` for some tools. The
PATH_WHITELIST exempts `/Users/yongqian/Desktop/Vessel/` paths from the regex
mask. The default DATA_DIR is `~/.vessel/...` = `/Users/yongqian/.vessel/...`
— **not in the whitelist**. So workspace paths at `/Users/yongqian/.vessel/workspace/<runId>/`
ARE pattern-redacted (they go through the `/Users/[^/\s]+/...` rule and don't
match the whitelist prefix). `metadata.cwd` thus becomes `***-redacted-xxxx***`
— this is over-redaction, not under-redaction. Not a leak.

BUT: when the user runs `pnpm vessel-core` from the Vessel dev tree, the
`spec.workspace` resolves under `/Users/yongqian/Desktop/Vessel/...`, which
IS whitelisted, so it WILL appear plaintext in stdout files. That's correct
because dev paths are by spec considered non-sensitive.

**The real BLOCKER**: `payload.message` (top level) ends with `.message`,
which IS in `REDACTED_FIELD_SUFFIXES` (line 51). For a CC system_init event
that has top-level `message` at all, this matches. But more critically, in
`tool_result` content like:
```json
{ "type": "user", "message": { "content": [{"type": "tool_result", "tool_use_id": "...", "content": "<file contents>"}] } }
```
- Path `payload.message.content[0].content` ends `.content` → masked. **GOOD**.

I traced through all the CC stream-json shapes I can recall and the suffix
list catches them. **Retracting this as BLOCKER**. Downgrading to:

#### MAJOR-4: stdout redaction is by suffix-match — adversarially fragile.

The list at `trace-redactor.ts:37-54` is a denylist by terminal field name.
Any future CC schema field that holds prompt content with a name not in the
list (e.g., `note`, `body`, `excerpt`, `chunk`, `paragraph`) would pass
through unredacted. The CC schema is owned by Anthropic and can change
between CLI versions (we ship Claude CLI v2.x today; v3.x might rename
`text` → `body`). **Recommendation**: invert the policy for CC stream-json
specifically — for any line read from `stdoutLines`, redact AGGRESSIVELY by
default (replace ALL string leaves longer than ~40 chars with redacted), and
allow-list short structural enums (type, role, model id, session_id, tool
name). This is consistent with trace-redaction-spec §3b's intent (when in
doubt, mask). Not a closeout blocker, but worth a `# TODO M0.5+` comment
+ retrospective entry.

#### BLOCKER-1 (corrected): `intent.received` trace event payload includes `text_len`, not `text`. **PASS** for that event.

`orchestrator.ts:84` writes `payload: { intent_id: intentId, text_len: input.text.length }`.
This is correct — only the length, not the prompt body. ✓

But **BLOCKER-1 now refocuses**: `writeIntent` (`session-store.ts:85-92`)
writes `text` (full user prompt) into `intents.text` SQLite column. AND
`writeSkillInvocation` (`session-store.ts:94-115`) writes `JSON.stringify(args.artifact)`
which for CodingArtifact contains `files: [absolute paths]`. The `memory.db`
itself is **NOT redacted**. Per Vessel design ("memory" is a feature; recall
needs the full prompt), this is intentional — the trace-redaction-spec
explicitly excludes memory.db from its scope (§5 mentions only trace files
and §7 acceptance only checks `instance/traces/...`).

But for the M0.5 acceptance test C-3 (`grep -q "user_prompt" instance/traces/<trace_id>/*.json`),
this is fine because no event payload carries `user_prompt`. The orchestrator
even aliases the path (`text_len` not `user_prompt`). ✓

**Net BLOCKER status for Focus 4**: NONE. Re-categorizing as MAJOR-4 above.

Wait — I missed something important on a re-read.

#### BLOCKER-1 (third read): `cli-runner-driver.ts:131-138` writes redacted lines to `stdoutPath` BUT does NOT enforce the 4 KiB inline limit.

trace-redaction-spec §5 mandates that oversized payloads spill to a separate
file (mode 0600), and the inline payload becomes a 200-char redacted
summary. The driver writes the FULL stream of CC stdout (potentially
megabytes of model output) to `<DATA_DIR>/traces/<trace_id>/<span_id>.stdout`
at mode 0600. **That file IS the spillover file** — there's no 4 KiB limit
because it's already the spill. ✓

But the file path `<span_id>.stdout` overlaps with `trace-writer.ts:53`
which ALSO writes to `<span_id>.stdout` for spilled trace event payloads.
If both happen for the same span_id, the trace-writer's spill OVERWRITES the
driver's stdout capture (or vice versa, depending on order). The driver
writes once at end-of-run; trace-writer writes whenever a payload exceeds
4 KiB. For the `skill.completed` event at `orchestrator.ts:116-126`, the
payload is `{ artifact_kind: 'coding' }` — tiny, no spill. So in practice
no collision today, but the convention is fragile.

**MINOR-1: collision risk between `cli-runner-driver.ts` stdout spillover and
`trace-writer.ts` payload spillover at the same `<span_id>.stdout` path.**
Suggestion: driver writes to `<span_id>.cli.stdout` or `<span_id>.driver.stdout`
to namespace.

A. **Update FRAMEWORK.md §2.2** to add `abortSignal: AbortSignal` to
   SkillContext, with a comment matching skill.ts:40-42 ("fires on SIGINT/SIGTERM").
   This is the simplest correct path.

B. **Bump some version marker.** There is no `interface schemaVersion` field
   for SkillContext (Skills don't yet declare their own contract version).
   AppManifest has `schemaVersion: 1` but that's the App-level manifest, not

Whether this is BLOCKER or MAJOR depends on whether 0B "froze" anything.
Given:
- FRAMEWORK.md is checked in (the spec).
- skill.ts is NOT yet committed (untracked).
- 0B closeout cross-review approved with the 4-field SkillContext.

this isn't a real BC concern. **Resolution**: MAJOR — update FRAMEWORK.md §2.2
in this M0.5 closeout patch. Don't ship M0.5 retro until FRAMEWORK reflects
the implementation.

## Out-of-focus findings (architect noticed)

### MAJOR-6: Echo prefix dispatcher is fragile.

`orchestrator.ts:48`: `if (text.trimStart().startsWith('echo '))`. This
silently classifies any user prompt that starts with the four characters
"echo " as an EchoSkill, NOT a coding task. So `pnpm vessel-core "echo this is
my homework about fibonacci"` returns `echoed: this is my homework about
fibonacci` instead of writing files. Fine for M0/M0.5 testing, but a real
trap once anyone uses Vessel for actual work. **Recommendation**: change
the prefix to `/echo ` (slash-prefixed, like CC's slash-commands), or
require a CLI flag `--skill=echo`. M0.5 acceptance text does not exercise a
real `echo`-ambiguous prompt, so this is technically meeting the bar.

### MAJOR-7: `cli-runner-driver.ts` always uses `permissionMode: 'bypassPermissions'`.

Line 112: `permissionMode: spec.permissionMode ?? 'bypassPermissions'`. This
is intentional (M1B will wire the permission router) and is documented in
the comment. But during M0.5, the Coding CLI runs with full filesystem
access — `cwd` is enforced (workspace-iso) but `Read`/`Write`/`Bash` tools
inside the CLI can see anywhere `claude` itself can. capability-coding's
manifest.yaml declares `permissionScope.paths: ${VESSEL_DATA_DIR}/workspace/**`
but per the manifest comment "(M1B 起 enforced；M0.5 仅声明)" — it's not
enforced. **Recommendation**: M0.5 retrospective must explicitly note "the
manifest paths field is not enforced; CC CLI inside the workspace has full
FS access via its bypass-permissions mode. M1B will wire enforcement." This
is acceptable for a personal-use kernel but must be visible.

### MINOR-2: `mapModel` model id strings.

`cli-runner-driver.ts:38-43`:
```typescript
case 'opus':   return 'claude-opus-4-7';
case 'sonnet': return 'claude-sonnet-4-6';
case 'haiku':  return 'claude-haiku-4-5';
```

Hardcoded model ids. When Anthropic ships claude-opus-4-8, this will need
manual update. Not a blocker for M0.5 (only uses Sonnet by default per
spec.model fallback at line 109). MINOR.

### MINOR-3: stdout line buffer holds entire stream in memory.

`cli-runner-driver.ts:97`: `const stdoutLines: string[] = [];` — pushed in
the `onMessage` callback. For long-running coding sessions (10k+ messages,
each maybe 1 KiB), this is up to 10 MB resident. trace-redaction-spec §5
mandates a 4 KiB inline limit per event; here we accumulate without
streaming to disk. **Recommendation**: stream-write each `onMessage` line
to the .stdout file directly with a stream writer, redacting per line.
Currently we hold all lines, redact at end, write at end. For M0.5 personal
use this is fine.

### MINOR-4: FakeCodingDriver workspace iso check is weaker than ClaudeCodeDriver.

`fake-coding-driver.ts:46-49`:
```typescript
if (!existsSync(spec.workspace)) {
  mkdirSync(spec.workspace, { recursive: true, mode: 0o700 });
}
const wsReal = realpathSync(spec.workspace);
```

No realpath-vs-realpath escape check. Comment at lines 43-45 acknowledges
this ("orchestrator already enforces the path"). But the orchestrator does
NOT enforce — it just calls `workspaceFor(runId)` which is a string join.
The actual enforcement is in the REAL driver. So FakeCodingDriver has
weaker guarantees than the real one. This means the integration test
(test-coding-driver.ts) does NOT exercise the workspace-escape path. **MINOR**:
add a workspace escape regression test that uses ClaudeCodeDriver with a
mocked `spawn` (to avoid burning a real `claude` invocation), or duplicate
the iso check in FakeCodingDriver and add an explicit "create symlink to
~/.ssh and verify the driver throws" test.
