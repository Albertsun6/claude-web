## BLOCKER（3 条，含 escalation-required）

### B-R1: 🚨 Secrets 扫描缺前置门槛（4 类硬触发命中）

**位置**：plan §「关键设计决策」未提；R-06 缓解策略写"git filter-repo 清理；扫描 commit 历史；移仓库前做 audit"——但**没说扫描发生在哪个阶段**。0-pre Acceptance 5 条里没有"secrets 扫描通过"。

**问题**：Eva 是用户自己的私有项目，commit 历史可能含 API key / 私人路径 / OAuth token。**fork-rename 为 Vessel 之前必须扫**——否则 secrets 跟着进 Vessel git 历史，未来开源分享时无法清理（除非 git filter-repo 重写历史，破坏所有引用）。

**4 类硬触发命中**：secrets 检测 → ADR-014 §「Escalation 触发器 / 硬触发」第 5 条要求**必停 owner 确认**。

**风险后果**：
- 高严重：Vessel 公开仓库 → 所有 commit 历史可被搜索 → secrets 暴露
- 中严重：即使保持私有，未来合作者可见

**建议**：
1. 把 secrets 扫描提前到 **0-meta-lite 阶段末尾**（在 0-pre 开始读 Eva 之前）
2. 加 0-pre Acceptance 第 6 条："`git log --all -p | gitleaks detect --no-banner` 扫 Eva 仓库无 high-severity finding"
3. 推荐工具：`gitleaks` / `trufflehog` / `git-secrets`，选其一在 ADR-014 增订段写明
4. 如有 finding → escalation 立即停 + 通过 ADR-013 rename strategy 决定是 git filter-repo 还是放弃 fork

### B-R2: ML worker 启动失败 fallback 未规划

**位置**：ADR-012 Consequences ② "Python worker 失败只降级对应 capability，不影响 core"；M1C-B Acceptance 第 3 条 "ML worker 失败时主进程标 memory capability unavailable"。

**问题**：ADR-012 假设 worker 已经能启动起来再失败。但**首次启动**有更多失败模式：
- fastembed 模型下载失败（网络问题 / 中国网络环境）→ R-02 已登记但缺验证
- Python 版本不兼容（用户的 Python 不是 3.10+）
- pip install 失败（依赖编译错误，比如 piper-tts 在 Apple Silicon 编译问题）
- `ml-workers/` 目录权限不对（用户 home 权限问题）
- venv 没建好（用户没装 uv 或 venv）

任何一个 → worker 永远启动不起来 → "降级"机制根本触发不了，主进程不知道是"失败"还是"还没启动"。

**风险后果**：M1C-B 实施时用户跑 `vessel-core memory search`，等 30 秒看不到结果——以为是 search 慢，实际是 worker 没启动。

**建议**：
1. ADR-012 Consequences 加 ④ 条：worker **首次启动失败模式** + 检测策略（health check / 启动超时 + 详细错误）
2. M1C-B Acceptance 加："ML worker 启动失败时 `vessel-core --health` 报具体原因（不是泛 'unavailable'）"
3. 0-pre RISKS R-11 缓解策略加："首次启动 health check + 错误归因到下载失败/Python版本/编译失败/权限/venv"

### B-R3: 🚨 Trace payload 脱敏白名单未定（隐私风险）

**位置**：plan §「关键设计决策 / 观测性 / Trace 字段」`payload | JSON | ✗ | 小负载；JSON-only / ≤ 4KB / 强制脱敏`。

**问题**：plan 说"强制脱敏"但**没定义脱敏规则**。CC CLI 输出常含：
- 用户 prompt 全文（个人信息 / 私人想法）
- 文件绝对路径（用户 home 路径泄露）
- API key（如果 prompt 里贴了）
- API responses（含敏感内容）

Trace 归档到 `instance/traces/<trace_id>/` 长期保存——**用户的 EVA 灵魂记忆变成永久日志**。

**4 类硬触发临近命中**：虽然这是用户本机数据，但若用户分享某次 trace 给开发者排查 bug，secrets 会跟着流出。

**风险后果**：
- 高严重：用户分享 trace 时无意泄露 secrets
- 中严重：trace 归档膨胀（CC CLI 输出大），磁盘占用失控

**建议**：
1. 0A FRAMEWORK 段必须定义**脱敏白名单 + 黑名单**：
   - 白名单：`event_type` / `component` / `status` / `duration_ms` / `trace_id` / `session_id` 总是明文
   - 黑名单：`payload.user_prompt` / `payload.cli_args.path`（除非在白名单路径）/ 任何匹配 `[A-Za-z0-9_-]{20,}` 的 token-like 字符串
   - 大输出：> 4KB → 切到 `artifact_refs`，文件级 mode 600
2. 0A 加 ADR-016（或扩 ADR-014）"Trace 脱敏规则"
3. M0 acceptance 加："trace 文件 mode 必须 600；payload 不出现 `user_prompt` 全文"

## MAJOR（4 条）

### M-R1: iOS bundle id 改名 → TestFlight 重审风险未充分登记

**位置**：plan §「M2-iOS Acceptance」`iOS bundle id 已改名 + TestFlight 验证通过`；R-09 缓解策略只说 "iOS bundle id 改名走 TestFlight 验证"。

**问题**：bundle id 改名 → Apple 把它当新 app → TestFlight 重新审核（2-3 天）；现有 TestFlight 用户必须重新安装；provisioning profile 重建。如果 Eva 现有 bundle id 是 `com.yongqian.claude-web` → Vessel 改成 `com.yongqian.vessel`，需要：
- App Store Connect 新建 app record
- 新 provisioning profile（开发 + ad-hoc + distribution）
- icon / display name 同步改
- 旧 TestFlight build 的现有用户无法 OTA 升级

**风险后果**：M2-iOS 阶段以为 1 天搞定，实际可能 3-5 天卡在 Apple 流程。

**建议**：
1. R-09 缓解策略细化：列具体步骤 checklist
2. 0-pre EVA_INVENTORY 列出当前 Eva iOS 完整配置（bundle id / display name / Team ID / cert / profile / icon resources）
3. M2-iOS Acceptance 加："改名前 Eva 旧 build 在 TestFlight 仍可用（不破坏现有体验）"

### M-R2: secrets 检测缺自动化机制（escalation 不可靠）

**位置**：ADR-014 §「Escalation 触发器 / 硬触发」第 5 条 `secrets 检测 → 必停 owner`。

**问题**：写了"必停"但**没说怎么检测**。手动每次 commit 前肉眼扫描必然漏。0-meta-lite 也没把工具写进去。

**风险后果**：第一次 secrets 漏到 commit 历史 → 永久污染。

**建议**：
1. 0-meta-lite 阶段就写一个 pre-commit hook 调 `gitleaks` 或 `git-secrets`（小工程量）
2. ADR-014 §「Escalation 触发器 / 硬触发」第 5 条改"`gitleaks` 或 `git-secrets` pre-commit hook 触发 → 必停"

### M-R3: 破坏性数据迁移检测缺 SOP

**位置**：ADR-014 §「硬触发」第 8 条 `schema 演进涉及 drop column / drop table / data loss → 必停`；ADR-006 Schema 演进策略。

**问题**：ADR-006 写"deprecated 字段保留 ≥ 1 个 minor 版本"，但**没说怎么 detect 一次 schema 改动是不是 drop**。Eva → Vessel 一次性数据迁移（在 0B 阶段跑）涉及表结构变化，谁在 review 时发现"这是 drop 不是 add"？

**风险后果**：用户 Eva 数据被静默丢失（如 inbox 表某字段被 drop），用户上次记录的 inbox 项消失。

**建议**：
1. ADR-006 加自动检测脚本：跑 `pnpm migrate:eva-to-vessel --dry-run` 后 diff 旧 schema vs 新 schema，报出所有 drop column / drop table / drop index
2. 0B Acceptance 加："dry-run 输出无 drop 操作，或 drop 操作有 owner approval 文件"

### M-R4: iOS Local Network 权限拒绝场景未规划

**位置**：plan §「M2-iOS / iOS 服务发现」mDNS 在 iOS Local Network 权限被拒时失败。

**问题**：iOS 14+ 跑 NWBrowser 必须在 Info.plist 申请 `NSLocalNetworkUsageDescription`，**用户首次启动时弹窗**——用户可能拒绝。拒绝后 NWBrowser 直接 fail，自动发现走不通。Acceptance 写"手填 IP fallback"但没说**怎么引导用户走到这一步**。

**风险后果**：用户拒绝权限 → NWBrowser 报错 → 用户不知道是权限问题，以为 app 坏了。

**建议**：
1. M2-iOS Acceptance 加："Local Network 权限被拒时 UI 显示具体原因 + 手填 IP 入口可见"
2. 加 RISKS R-12："iOS Local Network 权限拒绝率（缺数据，但企业 MDM / 隐私意识用户拒绝率不低）"
