### [BLOCKER-2] runMigrations 设计与 ADR-0015 不可调和（Lens: 架构可行性）

**Where**：`packages/backend/src/harness-store.ts` 第 63-77 行：

```ts
function runMigrations(db: Database.Database, fromVersion: number): void {
  const files = readdirSync(MIGRATIONS_DIR)
    .filter((f) => /^\d{4}_.*\.sql$/.test(f))
    .sort();
  for (const file of files) {
    …
    db.exec(sql);  // exec 内含 BEGIN/COMMIT 不可嵌套，靠 SQL 自身的 PRAGMA 推进 user_version
    …
  }
}
```

**Issue**：`fromVersion` 参数被传了但**完全没用**——每次启动都把 migrations/ 目录下**所有** `.sql` 文件重 exec 一遍。M-1 的 `0001_initial.sql` 全部用 `CREATE TABLE IF NOT EXISTS` 所以表面上 OK。但 ADR-0015 §3.2 明确允许（甚至期待）后续 migration 包含：

- `ALTER TABLE … ADD COLUMN`（minor bump 加字段）—— SQLite 的 ALTER 没有 IF NOT EXISTS 等价语法，第二次执行直接报 `duplicate column name`。
- 加 enum 值（rebuild table 模式）—— 不幂等。
- 数据 backfill / `INSERT INTO methodology …` —— 直接产 PK 冲突或重复 row。

注释里写着"靠 SQL 自身的 PRAGMA 推进 user_version"——但当前 0001_initial.sql 里 `PRAGMA user_version = 100` 是无条件赋值，不是条件判断；将来 0002_xxx.sql 即便也写 `PRAGMA user_version = 110`，`db.exec` 仍然不会 short-circuit 跳过 0001。`exec` 单调推进只防 user_version 倒退，不防 DDL 重复执行。

**Why blocker**：
1. 这是**生产数据安全风险**：第二次 migration 落地时（M0 / M1 加任何 DDL）后端启动失败或表结构损坏，且 `harness.db` 永远不会自动从 `harness-audit.jsonl` 救活（jsonl 没有完整 row state）。回滚靠手动重建 db。
2. 与 ADR-0015 §3.2 "每个 migration 必须是幂等的（CREATE TABLE IF NOT EXISTS 等）" 的措辞看似一致——但 ADR 说的是 "DDL 写法幂等"，runner 现在的实现是 "**忽略历史，每次全 exec**"，两者不是一回事。前者要求 SQL 作者注意；后者强制 SQL 作者把每条语句都写成幂等，这在 ALTER / 数据迁移上**不可能**。
3. 修起来必须改 runner 逻辑（加 `schema_migrations` 表 / 用 user_version 做 gate / 拆 up-down），M0 改最便宜，等到 0002 出现时改容易留 corrupted 中间态。

**Suggested fix**：典型 SQLite migration runner（任选其一）：

A. **user_version gate（最小改动）**：每个 migration 文件第一行写 `-- TARGET_VERSION = 110`，runner 解析后用 `db.pragma('user_version')` 检查；只在 `current < target` 时 exec，并在 SQL 末尾保留 `PRAGMA user_version = <target>;`。0001 改写时把 `PRAGMA user_version = 100` 移到文件末尾（已经是末尾附近，再确认）。

B. **schema_migrations 表**：runner 自建 `CREATE TABLE IF NOT EXISTS schema_migrations(file TEXT PRIMARY KEY, applied_at INTEGER)`，跑前 SELECT，跑后 INSERT；每个 migration 在事务里执行（`BEGIN; …; INSERT INTO schema_migrations …; COMMIT;`）。

A 更轻；B 与 ADR-0015 §3.1 "DB 全局 PRAGMA user_version" 兼容并加冗余。建议 A + 把 audit log 里 `op="migrate"` 写明从哪到哪。

（Cross-cut: 与 reviewer-cross 的 "irreversible operations" 镜头有交叉。主 lens 仍是架构可行性，因为问题在 runner 的设计而非具体 SQL 语句。）

## 四维评审

### 架构可行性

**评分（可选）**：3.5/5。整体方向正确，关键抽象稳定，但 runner 实现拖了后腿。

**站得住的设计**：
- 13 实体边界划得清晰，FTS5 contentless 模式 + 6 个触发器避免双写 (HARNESS_DATA_MODEL.md §1.3 + §1.10)。这套写法是 SQLite FTS5 教科书做法，触发器在 `0001_initial.sql:71-80, 232-241` 实测一致。
- `audit log = JSONL` / `business = SQLite` 的拆分（ADR-0010 §"为什么 audit log 不进 SQLite"）合理。理由 4 条全部成立——尤其"DB 损坏时审计日志可独立救活" 这条在 BLOCKER-2 触发的场景下就是救命稻草。
- `Artifact.storage in (inline, file)` + `size_bytes` + `superseded_by` + content-addressed `hash` 的组合是这份 schema 里最干净的部分。`CHECK ((storage='inline' AND content_text IS NOT NULL AND content_path IS NULL) OR …)` (`0001_initial.sql:219-222`) 这种约束 + `test-harness-schema.ts:90-106` 反例验证 = 标准实践。
- `idea_capture.processed_into_issue_id IS NULL` 偏索引 (`0001_initial.sql:96-97`) 直接对应 §0 #14 "30 秒内能存一条想法" 的 unprocessed 队列查询，针对性强。

**会变负债的抽象**：
- **`weight in (heavy/light/checklist)` 与 Stage.kind 的耦合写死在文档**（HARNESS_DATA_MODEL.md §1.5 文字段，DB 没有 CHECK）。文档说 "design / implement / test / review / release 是 heavy；strategy/compliance 默认 checklist"——但 schema 没强制。第一次有 Issue 想跑"轻量 spec"或"重量 strategy"时，要么改文档要么改默认值，会出现"文档说一回事、DB 一回事"的漂移。建议在 methodology 里下沉 `default_weight` 字段，Stage 创建时拷贝；不再让文档承诺 enum 映射。
- **`stage.{input,output,review_verdict}_artifact_ids_json` 三个 JSON 数组列**（`0001_initial.sql:132-134`）——和 §0 #9 "上下文严格管理" 起冲突。当 Context Manager 要按 stage 反查"这条 artifact 被哪些 stage 用过"，用 JSON 数组要全表 scan。建议拆中间表 `stage_artifact(stage_id, artifact_id, role)` —— 这是标准多对多。现在埋着，M1 跑 dogfood 要查报表时一定回来改。
- **`context_bundle ↔ task` 循环 FK** (HARNESS_DATA_MODEL.md §1.8 + `0001_initial.sql:148` 注释 "环依赖通过应用层 enforce")——能跑但应用层稍有疏忽就外键漂浮。Postgres 有 `DEFERRABLE INITIALLY DEFERRED` 解法，SQLite 没有。可以接受作为 M-1 妥协，但应该在 ADR-0010 里**明确写下"环依赖由 harness-store 写入序列保证"**，否则 M1 写 CRUD 的人会不知道这个不成文契约。

### 企业管理系统垂直贴谱性

**评分**：3.5/5。

- 13 实体里**唯一显式企业贴谱** 的是 `methodology.applies_to in (claude-web, enterprise-admin, universal)`（`0001_initial.sql:107`）和 plan §6.1 "spec 方法论企业字段必填段"。schema 自身**没有任何**业务实体 / 权限矩阵 / 审批流 / 报表口径的字段或表。
- 这本身**不是问题**——plan §0 已经说"方法论模板里专门写企业字段，schema 保持通用"，这是合理取舍：业务字段下沉到 `Artifact.kind in (spec, design_doc, ...)` 的内容里，DB 不耦合垂直。
- **真正的风险**：当企业 Issue 跑到 review/release Stage 时，需要从 spec/design 里抽出"哪个权限点改了""哪个审批流改了"喂给 reviewer，但 `Artifact` 是 markdown 全文，没有结构化抽取。HARNESS_PROTOCOL.md §2 把 `*_json` 扁平展开，**没有** "企业方法论字段" 的 typed schema。M2 跑 toy 企业后台时，reviewer 要么解析 markdown frontmatter（脆弱），要么 retrofit 一个 `enterprise_spec` 表（违反 §0 #11 "不引入新基础组件"）。
- **建议**：在 `Artifact` 上加一个 `metadata_json TEXT` 列（可选，default '{}'），约定 spec/design 类 Artifact 在 metadata 里塞 `{businessEntities: [...], permissionMatrix: {...}, approvalSteps: [...], reportSchemas: [...]}`。这是 minor bump（加列），M-1 现在就能加 schema 不打。结构由 methodologies/01-spec.md 的"企业字段必填段"定义；reviewer 解析 metadata 即可。
- 如果不加 metadata 列，至少要在 HARNESS_DATA_MODEL.md §2 实体关系图旁写一段 "为什么企业字段不进 schema"，帮 M2 reviewer 不再追这块。
- **会空跑的 Stage**：企业管理系统的 `release` Stage 在没有自动部署管道时基本是 stub（typically 是"用户手动 deploy 到 staging"+"产 changelog_entry Artifact"）。schema 上没 release_target 之类字段，靠 `Artifact.ref` 装 PR URL 撑着。可以接受作为 M-1 范围；M2 dogfood 时可能要再补。

### 风险遗漏

**评分**：3/5。已经识别的 18 条风险（HARNESS_RISKS.md）合理；本契约层面新增以下漏覆盖：

1. **migration 失败的回滚路径未定义**——`harness-store.ts:74` 注释说 "exec 内含 BEGIN/COMMIT 不可嵌套" 但没说 exec **没在外层事务里**。当某个 migration SQL 执行到一半失败（FTS5 trigger 创建报错等），DB 处于"表建了 trigger 没建"的中间态，下次启动 BLOCKER-2 提到的 re-exec 又会撞 "trigger already exists"。需要在 runner 里把每个文件包在 `db.transaction(() => db.exec(sql))()` 里。
2. **harness-audit.jsonl 写入失败处理未定义**——HARNESS_DATA_MODEL.md §1.14 说"DB 损坏时审计日志可独立救活"，但反过来"audit log 写盘失败时业务事务怎么办"没规定。如果 audit append 异步、业务先 commit，崩溃窗口里会丢审计；如果同步、append 失败时业务也得 rollback，那"无锁竞争"的优势就没了。建议 ADR-0010 §"为什么 audit log 不进 SQLite" 后加一段 "audit 失败处理：fail-open（log warn，业务继续）/ fail-closed 二选一"——和 permission-hook.mjs 的 fail-open 一致最自然。
3. **content-addressed Artifact 的 hash 碰撞虽然概率低但语义未定**——HARNESS_DATA_MODEL.md §1.10 写"先查 hash 命中即复用 row"。如果两个 Stage 上传相同内容，会复用同一行（id 不同？还是 id 也复用？schema 上 `id` 是 PK 不是 hash，`hash` 是普通索引允许重复）。需要明确：(a) hash 重复时复用旧 row 的 id（artifact 真共享）还是建新 id 同 hash（artifact 同源但每 Stage 独立）。当前 DDL `CREATE INDEX idx_artifact_hash ON artifact(hash)` 不是 unique，意味着允许重复——但文档说"自动去重"。冲突。
4. **FTS5 触发器在大批写入时的性能未估**——单 Issue 全链路写 5-15 个 Artifact，每条都过 `artifact_fts_ai`（含 `WHEN new.content_text IS NOT NULL`）。M2 dogfood 5 任务集 × 10 stage × 5 artifact 量级仍然 OK；但要在 Retrospective 里加一项"FTS5 写延迟"观察，以防 M3 量级膨胀。
5. **ADR-0015 §3.3 round-trip 不防"字段语义漂移"**——TS encode → Swift decode → Swift encode → TS decode deep-equal 只测**编码兼容**，不测语义。比如 `Stage.weight` 在两端都映射成 string 但 TS 端把 `'heavy'` 当业务级、Swift 端展示成本地化 label，round-trip 通过但 UI 行为分裂。这条不算 BLOCKER（M-1 还没到 UI），但 M0 实际上协议端要补一个 "enum 值锁"测试（HARNESS_PROTOCOL.md §1 已经把所有 enum 列出来了，可以直接生成 Swift 端枚举案例）。
