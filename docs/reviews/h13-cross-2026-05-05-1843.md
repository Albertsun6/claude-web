## Findings

### M1 [MAJOR] Hook runner bypasses `EvaConfigSchema`, so schema guarantees do not protect execution

**Where**: `scripts/eva-hook.mjs:62`, `scripts/eva-hook.mjs:69`, `scripts/eva-hook.mjs:75`; schema/tests at `packages/shared/src/eva-config.ts:45`, `packages/shared/src/eva-config.ts:47`, `packages/shared/src/__tests__/eva-config.test.ts:273`, `packages/shared/src/__tests__/eva-config.test.ts:290`
**Lens**: 正确性 / 安全
**Issue**: `eva-hook.mjs` directly `JSON.parse`s `eva.json` and reads `raw.worktrees?.find(...).hooks?.[hookName]` instead of validating with `EvaConfigSchema` or equivalent runtime checks.
**Why this matters**: H13's strict hook contract is currently enforced by schema tests, but not by the shell execution path. A malformed local `eva.json` can still be executed with overlong commands, wrong field types, invalid worktree entries, or empty commands treated as silent no-op. For a shell execution feature, the executor is the boundary that most needs the validation.
**Suggested fix**: Before lookup, validate enough of the config in `eva-hook.mjs`: `worktrees` must be an array; `name` / `path` must be strings; `hooks` must be a plain object with only the 4 valid keys; command must be string length `1..2000`. If importing compiled shared schema is practical, use `parseEvaConfig`; otherwise duplicate this minimal hook-specific validation locally.

### M2 [MAJOR] Full hook command is printed before execution, which can leak secrets into terminal logs

**Where**: `scripts/eva-hook.mjs:95`, `scripts/eva-hook.mjs:96`
**Lens**: 安全
**Issue**: The runner logs `cmd` verbatim before executing it.
**Why this matters**: Hook commands are likely to include inline env assignments or tokens during local automation experiments, and terminal output is persisted by developer tooling. A command like `CLAUDE_WEB_TOKEN=... curl ...` would leak immediately even before the command itself runs.
**Suggested fix**: Default to printing only hook name and cwd. Add `--verbose` for full command echo, or redact obvious `TOKEN|SECRET|PASSWORD|KEY` assignment patterns before printing.

### M3 [MAJOR] Hooks inherit the entire parent environment, including secrets and shell startup injection variables

**Where**: `scripts/eva-hook.mjs:98`, `scripts/eva-hook.mjs:101`, `scripts/eva-hook.mjs:102`
**Lens**: 安全 / Operational Risk
**Issue**: `env: process.env` passes all credentials and shell control variables into `bash -c`.
**Why this matters**: Since `eva.json` is git-tracked and hooks execute arbitrary shell, a bad hook can read inherited tokens. Separately, non-interactive `bash` honors variables such as `BASH_ENV`, so ambient environment can inject code before the configured hook command.
**Suggested fix**: Use a curated env by default: keep `HOME`, `PATH`, `USER`, `SHELL`, `TMPDIR`, and any explicitly allowed variables; always strip `BASH_ENV`, `ENV`, `SHELLOPTS`, and `CDPATH`. If full env pass-through is needed, make it opt-in with a visible flag.

### M4 [MAJOR] No dry-run or confirmation around destructive hook examples

**Where**: `WORKTREE_LOCK.md:24`, `WORKTREE_LOCK.md:25`, `scripts/eva-hook.mjs:17`, `scripts/eva-hook.mjs:21`
**Lens**: 安全 / 不可逆
**Issue**: Documentation shows `pre-remove` commands with `pkill` and `rm -rf`, while the runner explicitly has no dry-run and directly executes.
**Why this matters**: Manual trigger reduces risk, but `pre-remove` is precisely the lifecycle point where destructive cleanup is expected. A typo in `dataDir`, `$HOME`, or hook text can destroy the wrong local state.
**Suggested fix**: Add `--dry-run` before merge, or require `--yes` for `pre-remove` when the command contains high-risk tokens such as `rm -rf`, `pkill`, `killall`, or `git worktree remove`. At minimum, document a mandatory `pnpm eva:hook --dry-run pre-remove ...` workflow once the flag exists.

### m1 [MINOR] `pre-remove` fails when the worktree path is already gone, blocking late cleanup hooks

**Where**: `scripts/eva-hook.mjs:83`, `scripts/eva-hook.mjs:84`, `scripts/eva-hook.mjs:85`; author concern at `/tmp/h13-cross-prompt.md:437`
**Lens**: 正确性 / Operational Risk
**Issue**: The runner requires `entry.path` to exist for every hook, including `pre-remove`.
**Why this matters**: Cleanup hooks often need to run after a partial manual cleanup, for example to remove `CLAUDE_WEB_DATA_DIR` or kill a dev server after the worktree directory was already removed. Fatal exit `3` makes recovery harder.
**Suggested fix**: For `pre-remove`, if `entry.path` is missing, run from `REPO_ROOT` or `process.cwd()` and expose the missing path as env/context; for other hooks, keep the current fatal behavior.

### m2 [MINOR] `WORKTREE_LOCK.md` field list omits `hooks`

**Where**: `WORKTREE_LOCK.md:43`, `WORKTREE_LOCK.md:45`, schema at `packages/shared/src/eva-config.ts:98`, `packages/shared/src/eva-config.ts:99`
