## BLOCKER（2 条）

### B-P1: M0.5 强行重构 cli-runner.ts 违反 Eva 优先复用原则

**位置**：plan §「M0.5 — Coding Driver 重构」+ EVA_TO_VESSEL_MAPPING `cli-runner.ts → Coding Driver / 抽 submit_coding_task 接口`。

**问题**：Eva cli-runner.ts 已经在生产里跑通了所有 5 个集成挑战（非交互模式 / auth / stdout 解析 / 进程组终止 / 工作目录隔离）。plan M0.5 写"按 Vessel `submit_coding_task` 接口契约重构"——但**重构成本**：
- 改 Eva 现有调用方（routes / scheduler 调 cli-runner 的地方）
- 风险：破坏 Eva 已经验证的边界条件
- 收益：抽象出来一层接口（但 v0.1 只有 CC 一个 driver，cursor/codex 留 v1+）—— **YAGNI 信号强**

更轻的方案：**包装一层而不动 cli-runner.ts**——新建 `packages/backend/src/drivers/coding/index.ts` 只是个 façade：
```typescript
import { spawnClaudeCli } from '../../cli-runner';
export const submitCodingTask = (spec) => spawnClaudeCli(specToArgs(spec));
```
0 重构 cli-runner.ts，既得到接口抽象又保 Eva 已验证行为。

**工程后果**：plan M0.5 当前路径下，Eva 已经踩过的坑可能复发；FakeCodingDriver 录制回放也变复杂（要录两层抽象）。

**建议**：M0.5 改"包装层"而非"重构"。EVA_TO_VESSEL_MAPPING 里 cli-runner.ts 的重构动作改成"加 thin wrapper export `submitCodingTask`，不动 cli-runner.ts 内部"。

### B-P2: 0-pre EVA_INVENTORY 工作量被严重低估

**位置**：plan §「0-pre 产物 1 — `docs/notes/EVA_INVENTORY.md`」+ 0-pre Acceptance 第 1 条。

**问题**：EVA_INVENTORY Acceptance 写"覆盖 packages/backend / frontend / shared / ios-native 全部 src 文件 + 测试覆盖率"——这意味着：
- backend：~30+ TypeScript 文件（cli-runner / routes/* / scheduler / stores / notifications/* / heartbeat / 等）每个写一行职责 + 副作用
- frontend：~50+ 文件
- shared：~10 文件
- ios-native：~30+ Swift 文件
- 数据库 schema 全表 DDL：harness.db / inbox / sessions / projects / telemetry / heartbeat 至少 6 个表

**总工程量估计**：8-15 小时（不是 1-2 小时）。0-meta-lite 用户已经被告知 1-2 小时——但 0-pre 是接下来阶段，没标工作量大。

**工程后果**：0-pre 拖长，影响"立即进 0-pre"的承诺；用户看到工作量爆炸可能换 plan。

**建议**：0-pre Acceptance 改"覆盖**核心模块**（M0/M0.5/M1A/M1B/M1C 路径上的文件）"而非"全部 src"。frontend / ios 的非核心文件可以留 `coverage: not-yet-mapped`，进入 RISKS。

### M-P1: Eva 现有 protocol.ts 字段未先盘点

**位置**：plan §「目录结构」`packages/shared/src/protocol.ts # 沿用 Eva ClientMessage/ServerMessage`；EVA_TO_VESSEL_MAPPING 写"加 Soul Spec / Capability 装卸 / Trace event 消息"。

**问题**：Eva 现有 protocol.ts 已经有 `ClientMessage / ServerMessage / harness_event / fs_subscribe / session_subscribe / sdk_message / permission_reply / user_prompt` 等丰富的消息类型（盘点报告显示）。直接说"加 Soul / Capability / Trace 消息"前应先盘点 Eva 是否已经有可复用的字段或事件。

**工程后果**：可能重复设计已有消息类型（如 trace event 跟 harness_event 部分重叠）。

**建议**：0-pre EVA_INVENTORY 单独一节列出 Eva protocol.ts 现有所有消息类型 + 字段；EVA_TO_VESSEL_MAPPING 增加一行"protocol.ts 字段复用清单"，明确哪些 Vessel 新增类型可借用现有字段。

### M-P4: 命令格式 `pnpm vessel-core "echo hi"` 在 Eva 仓库下不工作

**位置**：plan 多处 acceptance 用 `pnpm vessel-core "..."` 格式。

**问题**：Eva 现有 `package.json`（pnpm workspace 根）跑的是 Hono HTTP server（dev:backend）和 frontend。`vessel-core` 不是 Eva 现有的 npm script。要让 `pnpm vessel-core "echo hi"` 工作，需要：
- 改 root `package.json` 加 `"scripts": { "vessel-core": "node packages/backend/dist/cli.js" }`
- 或 backend `package.json` 里加 `bin` 字段
- 或用 `pnpm --filter @vessel/backend exec vessel-core ...`

**工程后果**：M0 第一个 acceptance 直接卡住——"`pnpm vessel-core "echo hi"` 退出码 0"。

**建议**：0A FRAMEWORK 锁定 CLI 入口实现方式 + 在 0B Acceptance 加 `pnpm vessel-core --version` 退出码 0。
