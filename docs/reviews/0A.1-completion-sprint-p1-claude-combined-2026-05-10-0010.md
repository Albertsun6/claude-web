## MAJOR（4 条）

### 🏛️ M-A1: A7 Trace 协议双 schema 命名空间会引入维护负担

**Where**: `FRAMEWORK.md` §5 + §5.5/5.6/5.7

**Issue**: v0A.1 选择"v0.1 保留 12 字段 + 加 OTEL GenAI 兼容映射 + 承诺 v1+ pivot"是稳妥做法，但**两套字段命名同时维护**（Vessel 12 字段 + OTEL `gen_ai.*`）会让：
- M0 实施 trace.ts 时：写两遍 / 写映射器
- 文档 / 测试 fixture 双份
- owner 调试时困惑（"trace 输出哪个 schema？"）

**Why MAJOR**: 不阻塞 0B（仅 stub），但 M0 实施时是真痛点。

**Suggested fix**: M0 实施时**直接走 OTEL 命名**（不保留 Vessel 12 字段命名），但**保留兼容性测试**——OTEL trace 反向能解析出 Vessel 概念字段（event_type / component / 等通过组合 `gen_ai.*` 字段恢复）。当前 v0A.1 §5.5 兼容映射表是好的，但 §5（12 字段 schema）应该改成"v0.1 实现时实际只落 OTEL 字段；12 字段是逻辑视图"。

### ⚙️ M-P1: A6 端口分配 11435/11436/11437 硬编码

**Where**: `FRAMEWORK.md` §4.1 表格 / `ADR-012` §6 隐含

**Issue**: 写死 3 个固定端口（11435 = embedding / 11436 = whisper / 11437 = piper）。问题：
- 端口冲突时 vessel-core 启动失败（owner 已经在 11435 跑了别的服务）
- 跨机器（macOS / Linux）端口规则差异
- v1+ 加新 ML worker（如 vision encoder）继续硬编码扩号（11438/11439...）会快用完私域

**Suggested fix**: §4.1 改成"**默认端口 + 可配置 + 动态分配**"：
- vessel-core 启动时从 `~/.vessel/config.toml` 读 `[ml_worker.embedding] port = 11435`（默认）
- 如端口占用 → 自动尝试 +1 至 +10 → 都占用 escalation owner
- worker URL 通过 vessel-core 进程内 service registry 暴露给 caller（不让 caller 硬编码）

### 🛡️ M-R1: A6 HTTP loopback 暴露生命周期管理新风险

**Where**: `FRAMEWORK.md` §4 + ADR-012 §6

**Issue**: 从 stdio JSON-RPC pivot 到 HTTP loopback 引入新风险点（**stdio 简单天然有的某些保证，HTTP 没有**）：
- **僵尸进程**：HTTP server 跟父进程没有 stdin/stdout 管道；父进程死了 worker 不会自动 EOF / 退出
- **认证缺失**：loopback 不出本机但**同机器其他用户/进程**可以连 127.0.0.1:11435（macOS 多用户场景）→ 任何本机进程能消耗 ML worker 资源
- **优雅退出**：HTTP `POST /shutdown` endpoint 必须实现；不实现就只能 SIGTERM 进程组（但 uvicorn 主线程吃 SIGTERM 才退）

**Suggested fix**:
- §4.4 加："worker 必须实现 `POST /shutdown` endpoint；vessel-core 退出时先调 shutdown 等 5s 再 SIGTERM"
- §4.4 加：**僵尸进程检测** —— vessel-core 主进程定期 ping `GET /health`；如 vessel-core 死后 worker 还活着（heartbeat 未续期），worker 自动退出（`/health` heartbeat 接收方实现）
- §4 加 "Auth"段：worker 校验请求来源 PID == vessel-core 的 PID（通过 `/proc/self/status` 或 macOS 等价 API），拒绝其他进程
- 或简化：worker 用 **Unix domain socket（`unix:///tmp/vessel-embedding.sock`）** 而非 TCP loopback —— 同样 OpenAI API 兼容（fastapi 支持 UDS），但天然只能本进程组访问

### 🏛️ M-A2: A1 soul.md 4 sibling 与 ADR-005 INSTANCE 隔离 + ADR-013 改名 strategy 联动

**Where**: `FRAMEWORK.md` §6.1 + ADR-005 + ADR-013

**Issue**: A1 把 soul.md 拆成 4 sibling 文件（SOUL.md / STYLE.md / SKILL.md / MEMORY.md），但：
- ADR-005 INSTANCE 隔离的 `.example` 模板从单文件改成 4 文件（每个都需 `.example`）
- ADR-013 改名 strategy 没说 4 sibling 文件名是否要带 vessel 前缀（避免与可能的其他工具冲突）
- M2-Soul `vessel init` 引导现在要 walk 用户过 4 个文件 vs 1 个文件，UX 复杂

**Suggested fix**: 加 ADR-018 或在 ADR-005 增订段：
- 锁定 4 sibling 文件名（**不**加 vessel- 前缀，沿用 OpenClaw 命名习惯：SOUL.md / STYLE.md / SKILL.md / MEMORY.md）
- `.example` 模板：每个 sibling 单独 `.example`
- `vessel init` 引导：默认 clone Soul Template（4 文件全套）；强制改 ≥ 1 字段（任意 sibling 内）
- M2-Soul Acceptance 加："`vessel init` 后 `instance/{SOUL,STYLE,SKILL,MEMORY}.md` 4 文件存在"

### 🛡️ m-R1: HTTP loopback 端口审计

**Where**: `FRAMEWORK.md` §4.1 默认端口

**Issue**: 11435 / 11436 / 11437 是 Vessel 自取的；**没核对是否冲突已知服务**：
- 11434 是 Ollama 默认（确认 Vessel 不冲突 ✅）
- 11435+ 区段没正式 IANA 注册（私域 ephemeral 49152-65535 才规范）

**Suggested fix**: §4.1 加注"端口选择避开 IANA registered（< 49152）+ 常见服务（11434 Ollama / 8080 / 3000 / 5000 / 等）；11435-11437 是 Vessel 自取，未来如冲突按 M-P1 fallback 流程"。

### 🏛️ m-A1: ADR-012 §6 接口契约跟 FRAMEWORK §4.2 略不同步

**Where**: `ADR-012` §6（旧 stdio JSON-RPC 接口）+ `FRAMEWORK.md` §4.2（新 HTTP client 接口）

**Issue**: ADR-012 §6 仍写 `embeddingClient.embed(texts) -> number[][]`，没标 v0A.1 修订；FRAMEWORK §4.2 已经详细写。读者会困惑哪份是真。

**Suggested fix**: ADR-012 §6 加"v0A.1 修订（A6 2026-05-10）：HTTP client 接口详见 [FRAMEWORK §4.2](../../design/FRAMEWORK.md)；本 ADR 接口签名仅作 reference，以 FRAMEWORK 为准"。

### 🏛️ m-A2: SoulSpec 4 sibling 兼容性段（FRAMEWORK §6.4）"实际无生产数据"假设可能误判

**Where**: `FRAMEWORK.md` §6.4

**Issue**: 写"v0A.1 之前如有单文件 soul.md（M2-Soul 还没实施，所以**实际无生产数据**），M2-Soul 实施时直接走 4 sibling"——假设 owner 在 0-meta-lite / 0-pre / 0A 期间没自己写过 soul.md。但**owner 在自己机器上随便写 soul.md 测**的可能不为零。

**Suggested fix**: §6.4 加："如 owner 在 0A.1 之前已有 `instance/soul.md` 单文件草稿，M2-Soul `vessel init` 检测到时执行一次性迁移：parse 单文件 → 拆字段 → 写 4 sibling → backup 旧文件到 `instance/soul.md.legacy`"。低成本 / 防数据丢失。
