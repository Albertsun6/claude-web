## Findings

### M1 [MAJOR] ID 格式契约自相矛盾：注释说 UUIDv4，schema 和 fixtures 实际是 opaque/prefixed string

**Where**: `packages/shared/src/harness-protocol.ts` 命名约定 `ID UUIDv4 字符串`; all `id` / `projectId` / `stageId` fields; fixtures such as `project.json:id = "proj-01HJK5X8Y2QMR3N4P9ABCDEF01"`  
**Lens**: 跨端对齐 / 不可逆  
**Issue**: wire 契约写明 ID 是 UUIDv4，但 TS Zod 只用 `z.string()`，Swift 只用 `String`，fixtures 又使用 `proj-...` / `iss-...` / `stage-...` 这类 prefixed ULID-ish ID。  
**Why this is a major**: ID 格式一旦被后端、iOS cache、DB FK、artifact path、audit log 写入真实数据后很难改。现在 contract reader 会以为可以发 UUIDv4，但 fixture reader 会以为应该发 prefixed ID。  
**Suggested fix**: 二选一并统一三端：  
- 如果要 opaque prefixed ID：把协议注释从 “UUIDv4 字符串” 改成 “opaque stable string, recommended `<type>-<id>`”，并给每类 ID 是否允许前缀写清楚。  
- 如果真要 UUIDv4：TS 用 `z.string().uuid()`，fixtures 全部改成 UUIDv4，Swift 侧至少在测试中验证 UUID decode。

### M2 [MAJOR] `HarnessEvent.stage_changed.status` 没有复用 `StageStatus` 枚举，事件可携带非法状态

**Where**: `packages/shared/src/harness-protocol.ts` `HarnessEventSchema` stage_changed branch field `status: z.string()`; `packages/ios-native/Sources/ClaudeWeb/HarnessProtocol.swift` `case stageChanged(stageId: String, status: String)`  
**Lens**: 正确性 / 跨端对齐  
**Issue**: Stage DTO 的 status 被 `StageStatusSchema` / `StageStatus` 约束，但 WS event 的 `stage_changed.status` 放宽成任意字符串。  
**Why this is a major**: UI 可能先收到 WS event 再刷新 DTO；如果后端发了拼错或未来状态，TS 端会 parse 成功，Swift 端也会 decode 成功，错误会进入运行态而不是在边界失败。  
**Suggested fix**: TS 改成 `status: StageStatusSchema`；Swift 改成 `case stageChanged(stageId: String, status: StageStatus)`，decode/encode 都使用 `StageStatus`。

### M3 [MAJOR] 跨端 round-trip 不变量没有真正覆盖 Swift 端

**Where**: `packages/shared/src/__tests__/harness-protocol.test.ts`; `packages/ios-native/Sources/ClaudeWeb/HarnessProtocol.swift` `AnyCodable` and custom `HarnessEvent` Codable  
**Lens**: 跨端对齐  
**Issue**: TS fixtures 有 Zod parse → JSON stringify deep-equal 测试，但 Swift 端只是注释说 “M-1 范围内人工抽样验证”。本次 Swift 里有两个高风险点：`AnyCodable` 自定义递归 JSON，以及 `HarnessEvent` 手写 discriminated union encode/decode。  
**Why this is a major**: 当前 review 的核心目标就是 TS Zod ↔ Swift Codable ↔ fixtures 对齐。没有 Swift fixture 测试时，Swift 可选字段、省略字段、AnyCodable 数字类型、event union 字段名都可能静态看起来对齐但实际 round-trip 失败。  
**Suggested fix**: M-1 就加最小 XCTest 或 Swift script：读取同一批 `packages/shared/fixtures/harness/*.json`，decode 到对应 Swift DTO，re-encode 后 JSON deep-equal。至少覆盖 `artifact-inline.json`、`artifact-file.json`、`audit-log-entry.json`、`harness-event.json`。

### m1 [MINOR] Zod 对 epoch ms 只约束 int，没有约束非负

**Where**: `createdAt`, `updatedAt`, `startedAt`, `endedAt`, `capturedAt`, `approvedAt`, `ts` fields in `packages/shared/src/harness-protocol.ts`  
**Lens**: 正确性  
**Issue**: 协议说明时间戳是 epoch ms 数字，但 schema 允许负整数。Swift `Int64` 同样允许负数。  
**Suggested fix**: TS 时间戳字段统一用一个 helper，例如 `EpochMsSchema = z.number().int().nonnegative()`；Swift 若不做 runtime validation，至少让 fixture/test 覆盖非法负值在 TS 边界被拒绝。

### m2 [MINOR] `ReviewVerdict.dimensions` 允许任意 number，没有沿用 score 的 0..5 范围

**Where**: `packages/shared/src/harness-protocol.ts` `ReviewVerdictDtoSchema.dimensions: z.record(z.number())`; Swift `dimensions: [String: Double]`  
**Lens**: 正确性 / 跨端对齐  
**Issue**: 总分 `score` 有 `min(0).max(5)`，但分项 `dimensions` 没有限制，可以出现 `-100` 或 `999`。  
**Suggested fix**: TS 改为 `z.record(z.number().min(0).max(5))`；Swift 侧若保持 plain `Double`，通过 fixture negative case 或 decode 后 validation 补齐。

### m3 [MINOR] Artifact hash 契约没有校验 `sha256:<64 hex>` 格式

**Where**: `packages/shared/src/harness-protocol.ts` `ArtifactDtoSchema.hash: z.string()`; fixtures `artifact-inline.json.hash`, `artifact-file.json.hash`  
**Lens**: 正确性 / 安全  
**Issue**: fixtures 使用 `sha256:<hex>`，但 schema 接受任意字符串。  
**Suggested fix**: TS 用 regex 锁定 `^sha256:[a-f0-9]{64}$`。Swift 可以继续用 `String`，但 Swift fixture test 应覆盖合法格式；真正路径落盘前后端也应再验证一次。
