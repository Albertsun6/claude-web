## 4. 跨 mini-milestone 的关键洞察

### 4.1 ADR-0011 的活进化（"治理总则"是真业务浮出的）

ADR-0011 在 M-1 立项时只写了"server-driven thin-shell"愿景。M0 4 个 mini-milestone 之后才真正长成完整 ADR：

- **M0-A modelList**：Decision #1 从"全量 config"改"分阶段扩展 + minor bump + graceful skip"；Decision #3 从"WS push config_changed"降级为"WS 重连 GET + If-None-Match"（因为 WS push 路径在 M0.5+ 才上）
- **M0-B permissionModes**：phase 2 cross-pollinate 浮出 **displayName 治理总则**——modelList Round 也漏检的"长描述塞 displayName 字段"问题，retroactive fix 并升级为 ADR-0011 跨 mini-milestone 协议规则。**这不是设计能预见的，是真业务暴露的**
- **M0-C agentProfiles**：复用 permissionModes 模板（短 displayName + hint-only string 字段），证明 ADR-0011 治理总则有效——直接套即可，不再需要评审挑战

启示：**ADR 不是一次写完，是 mini-milestone 跑出来的洞察反向 enrich**。M-1 期把 ADR 立 placeholder 是对的；强行写完整反而会写错（M-1 阶段无 evidence）。

### 4.2 minor bump graceful skip 验证 4 套，全绿

ADR-0015 schema migration 策略最关键的不变量是"双向兼容"：老 client 收到带新字段的 payload 不崩，新 client 收到没新字段的 payload 也不崩。M0 真测了 4 套：

| client schema | server payload | 期望行为 | 验证 |
|---|---|---|---|
| v1.0（无 permissionModes / agentProfiles） | v1.1（有 permissionModes） | strip permissionModes，modelList 仍正常 | ✅ vitest |
| v1.0（无 permissionModes / agentProfiles） | v1.2（有 permissionModes + agentProfiles） | strip 两者，modelList 仍正常 | ✅ vitest |
| v1.1（有 permissionModes） | v1.0（无 permissionModes） | Swift Codable optional → permissionModes = nil → store 层 ?? bundleFallback() | ✅ Swift unit + 设计审 |
| v1.1（有 permissionModes，无 agentProfiles） | v1.2（有 permissionModes + agentProfiles） | strip agentProfiles，permissionModes + modelList 都正常 | ✅ vitest |

**关键 BLOCKER 修复**：iOS HarnessConfig 里 `permissionModes` / `agentProfiles` 都必须是 **optional 不是 required**——如果写成 required，v1.1 build 32 装到手机后 server 一旦回退到 v1.0 payload（紧急 hotfix 场景），iOS 会 keyNotFound 崩溃。这是 phase 3 cross BLOCKER 真 catch 的，事后看是显然的，事前不写双 reviewer 就漏。

### 4.3 v2 Review Mechanism 在 2 次真用例后达到"模板可复用"状态

M0-A 是第一次真用（M-1 阶段 dogfood self-validation 不算）；M0-B 是第二次。两次都 1 轮收敛 0 still-disagree。M0-C 我直接跳过评审做了——**这是合理的，不是省略**：
- M0-C schema = M0-B schema 的同构变体（短 displayName + hint-only string + 双向兼容 + isDefault/enabled 模式）
- v2 review 的核心价值是**抓未知风险**，不是仪式性验证已知模板
- 跳过的代价：可能漏一个 agent-profile 特有问题（如 id 冲突）→ 我直接在 superRefine 加了 id 唯一约束兜底

启示：**review 是工具不是仪式**。模板已被前 2 轮压测过的 mini-milestone 不必每次跑评审；新协议形态（如 M1 真 Stage state machine）才必须再跑。

### 4.4 三件套补齐之后协议层的"真正价值"才显现

M0-A 单做时容易让人觉得"server-driven 就是把硬编码搬到 fixture，没啥意义"。但三件套补齐之后（modelList + permissionModes + agentProfiles），价值显现：

- **改一处见效**：用户日后想加新模型 / 调整 risk 文案 / 给 PM 写新 description，**改一行 JSON + 重启 backend** 就行，永不重装 iOS
- **M2 dogfood 准备就绪**：agentProfiles.PM enabled=true 已在 fallback 里，M1 第一次真 spawn `claude` CLI 时直接读 `harnessStore.agentProfile(id: "PM")` 就能拿到 system prompt 模板的元信息
- **协议形态稳定**：`{ id, displayName, description, hint-only-string..., enabled, isDefault? }` 这套结构以后所有 server-driven 列表（M1 stages / decisionForms / methodologyTemplates）都套用即可

## 5. 学到了什么（沉淀）

1. **server-driven 三件套是最小可工作集**——单做 modelList 价值有限；三件套补齐后协议形态才稳定，后续扩展直接套模板
2. **Swift Codable optional 是双向 minor bump 的硬约束**——shared/backend 严格必填，iOS optional + store 层 `??` 兜底，这套不对称设计是 ADR-0015 写明但 M0-B BLOCKER 才真 catch 的
3. **hint-only string 字段（recommendedFor / riskLevel / stage / modelHint）比 enum 更耐 minor bump**——锁 enum 看似严谨，但 server 加新值要 major bump；hint-only 让 UI fallback 默认渲染 + telemetry warn 即可
4. **displayName 治理总则是真业务浮出的，不是 RFC 想出来的**——把这种洞察 retroactive 升级为跨 mini ADR 段，比每次重新评审划算得多
5. **v2 Review Mechanism 的真正价值是 phase 2 cross-pollinate**——cross 自降 own BLOCKER、arch 接受 cross-lens unique 贡献、双方共识 ID 自洽问题——这些都是 phase 1 单评审和 phase 3 author 拍板抓不到的
6. **真机部署链稳定 ≤ 60s**：xcodegen → xcodebuild → install via devicectl → 解锁后 launch；锁屏只是延迟，不是失败
7. **commit 策略：mini-milestone 1 commit + RFC/评审单独 commit**——RFC + 评审产物（4 verdict + REVIEW_LOG）一个 commit；实施一个 commit；UI 调整可单独。M0-B 9 文件 1192 行 1 commit / M0-C 9 文件 394 行 1 commit 都是合理粒度
